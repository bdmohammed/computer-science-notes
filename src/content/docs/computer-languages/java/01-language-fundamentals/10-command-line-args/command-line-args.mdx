## 💻 Command Line Arguments in Java — Complete Guide

> **Command Line Arguments** are values passed to the program **at runtime** from the terminal — before the program starts executing.

---

## 📌 How It Works

```java
public static void main(String[] args)
//                       ↑
//              receives command line arguments
```

```text
Terminal:
java MyClass Alice 25 true

         args[0] = "Alice"
         args[1] = "25"
         args[2] = "true"
         args.length = 3
```

<Aside type="note">
**Key Rule**: Arguments are **space-separated** by default. Use **quotes** to pass values containing spaces as a single argument.
</Aside>

---

## 🔬 In-Depth — How Arguments Flow

```text
You type in terminal:
java MyClass Hello 99
        ↓ JVM receives
String[] args = {"Hello", "99"}
        ↓ passed directly to
public static void main(String[] args)

⚠️ Everything is received as STRING
   even if you pass 99 → it's "99" not int 99!
```

<Code lang="java" title="Visualizing argument reception" code={`public static void main(String[] args) {
    // JVM creates the array BEFORE main() runs
    // You cannot change args.length, but you CAN modify contents:
    args[0] = "Modified";  // ✅ Valid — array is mutable
    // args = new String[5]; // ✅ Also valid — reassign reference
}`} />

## 💻 Basic Example

<Code lang="java" title="Iterating through arguments" code={`class MyClass {
    public static void main(String[] args) {

        System.out.println("Total args: " + args.length);

        for (int i = 0; i < args.length; i++) {
            System.out.println("args[" + i + "] = " + args[i]);
        }
        
        // Modern style: enhanced for-loop
        int idx = 0;
        for (String arg : args) {
            System.out.println("Arg " + idx++ + ": " + arg);
        }
    }
}`} />

```text
Run:  java MyClass Alice 25 Engineer

Output:
Total args: 3
args[0] = Alice
args[1] = 25
args[2] = Engineer
```

## 🔄 Type Conversion — Always String First
All command line arguments arrive as `String`. Convert explicitly to other types.

<Code lang="java" title="Parsing different types" code={`public static void main(String[] args) {
    // Always check length before accessing indices!
    if (args.length < 4) {
        System.err.println("Usage: java App <name> <age> <salary> <flag>");
        return;
    }

    String name   = args[0];                          // "Alice"   → String ✅
    int    age    = Integer.parseInt(args[1]);        // "25"      → 25 int ✅
    double salary = Double.parseDouble(args[2]);      // "99.5"    → 99.5 double ✅
    boolean flag  = Boolean.parseBoolean(args[3]);    // "true"    → true boolean ✅
    
    System.out.printf("%s, age %d, salary $%.2f, active: %b%n", 
                      name, age, salary, flag);
}`} />

<Aside type="danger">
⚠️ **NumberFormatException Trap**: Passing non-numeric strings to `parseInt()`/`parseDouble()` crashes at runtime.
```java
java MyClass Alice abc
// Integer.parseInt("abc") → 💥 NumberFormatException!
```
**Always validate or use try-catch** before parsing.
</Aside>

<Code lang="java" title="Safe parsing with validation" code={`public static void main(String[] args) {
    if (args.length < 2) {
        System.err.println("Need at least name and age");
        return;
    }
    
    String name = args[0];
    int age;
    
    try {
        age = Integer.parseInt(args[1]);
    } catch (NumberFormatException e) {
        System.err.println("Age must be a number, got: " + args[1]);
        return;
    }
    
    System.out.println(name + " is " + age + " years old");
}`} />

## ⚠️ Common Mistakes
<CardGrid>
  <Card title="Mistake 1 — Accessing without checking length" icon="error">
```java
    public static void main(String[] args) {
        System.out.println(args[0]); // 💥 ArrayIndexOutOfBoundsException!
        // If user runs: java MyClass (no args)
    }
    
    // ✅ Fix — always check length first
    if (args.length > 0) {
        System.out.println(args[0]);
    }
```
  </Card>
  
  <Card title="Mistake 2 — Checking for null" icon="error">
```java
    public static void main(String[] args) {
        if (args == null) { }      // ❌ Never null! JVM always creates array
        if (args.length == 0) { }  // ✅ Correct check for no arguments
    }
    // args is ALWAYS a valid String[] — empty if no args passed
```
  </Card>
  
  <Card title="Mistake 3 — Spaces in arguments" icon="error">
```text
    java MyClass Hello World   
    // args[0]="Hello", args[1]="World" → 2 arguments!
    
    java MyClass "Hello World" 
    // args[0]="Hello World" ✅ → 1 argument with space
    
    // Windows CMD alternative:
    java MyClass "Hello World"
    // Linux/Mac alternative:
    java MyClass 'Hello World'
```
  </Card>
</CardGrid>

## 🎯 Interview Cheat Sheet

<CardGrid>
  <Card title="Q: What is the type of command line arguments?" icon="information">
    **Always `String[]`** — even numbers, booleans, or dates arrive as Strings.  
    You must explicitly parse them: `Integer.parseInt()`, `Boolean.parseBoolean()`, etc.  
  </Card>
  
  <Card title="Q: What if no arguments passed — is args null?" icon="approve-check">
    **NO ❌** — `args` is **never null**.  
    It's an **empty String array** → `args.length == 0`.  
```java
    // java MyClass (no args)
    // Inside main: args != null, but args.length == 0
```
  </Card>

  <Card title="Q: Can we pass spaces in a single argument?" icon="approve-check">
    **YES ✅** — wrap in quotes:
```text
    java MyClass "Hello World" → args[0] = "Hello World"
    java MyClass 'Path/To/File' → args[0] = "Path/To/File"
```
  </Card>

  <Card title="Q: What is args.length when no args passed?" icon="information">
    **`0`** — an empty array, not `null`.  
```java
    // java MyClass
    // args = new String[0]; // internally
    System.out.println(args.length); // 0
```
  </Card>

  <Card title="Q: Can we change args inside main?" icon="approve-check">
    **YES ✅** — it's just a mutable `String[]` array.
```java
    args[0] = "New Value";  // ✅ Valid — modifies array content
    args = new String[5];   // ✅ Valid — reassigns reference
    // But note: changing args has no effect outside main()
```
  </Card>

  <Card title="Q: Can we pass null as an argument?" icon="error">
    **NO ❌** — you cannot pass literal `null` from terminal.  
    However, you can pass the **string** `"null"`:
```text
    java MyClass null
    // args[0] = "null" (the 4-character string), not Java null reference
```
  </Card>
</CardGrid>

## 🧩 DSA Angle
<CardGrid>
  <Card title="Pattern: Passing test input via CLI" icon="rocket">
    Quickly test algorithms with different inputs without modifying code:
```java
    // java Solution 5 3 → n=5, k=3
    public static void main(String[] args) {
        if (args.length < 2) {
            System.err.println("Usage: java Solution <n> <k>");
            return;
        }
        int n = Integer.parseInt(args[0]); // 5
        int k = Integer.parseInt(args[1]); // 3
        System.out.println(solve(n, k));
    }
```
  </Card>
  
  <Card title="Pattern: Passing array elements" icon="shield">
    Convert CLI args directly into arrays for algorithm testing:
```java
    // java Solution 1 2 3 4 5
    public static void main(String[] args) {
        int[] arr = new int[args.length];
        for (int i = 0; i < args.length; i++) {
            arr[i] = Integer.parseInt(args[i]);
        }
        // arr = {1, 2, 3, 4, 5} ✅
        System.out.println(maxSubarraySum(arr));
    }
```
  </Card>

  <Card title="Pattern: Flag-based test selection" icon="backspace">
    Use arguments to toggle test cases or modes:
```java
    public static void main(String[] args) {
        boolean verbose = false;
        int testId = 1;
        
        // Parse optional flags
        for (String arg : args) {
            if (arg.equals("-v")) verbose = true;
            else if (arg.matches("\\d+")) testId = Integer.parseInt(arg);
        }
        
        runTest(testId, verbose);
    }
    // java Solution 3 -v  → runs test 3 with verbose output
```
</Card>
</CardGrid>

<Aside type="tip">
**Competitive Coding Pro Tip**: Many online judges (Codeforces, LeetCode CLI) don't use `args` — they read from `stdin`. But for local testing, `args` is faster than typing input repeatedly. Design your solution to support both:
```java
// Fallback to stdin if no args provided
int n = args.length > 0 ? Integer.parseInt(args[0]) 
                        : new Scanner(System.in).nextInt();
```
</Aside>

## 🏢 Real-Life SDE Usage

<CardGrid>
  <Card title="Spring Boot — auto-parsing CLI config" icon="rocket">
    Spring Boot automatically binds `--key=value` args to configuration:
```java
    @SpringBootApplication
    public class App {
        public static void main(String[] args) {
            SpringApplication.run(App.class, args);
        }
    }
    // java -jar app.jar --server.port=9090 --spring.profiles.active=prod
    // ↑ Spring reads these from args[] and overrides application.properties ✅
```
  </Card>
  
  <Card title="Environment switching via args" icon="shield">
    Simple pattern for deploying to different environments:
```java
    public static void main(String[] args) {
        String env = args.length > 0 ? args[0] : "dev";
        
        switch (env) {
            case "prod" -> connectToProdDB();
            case "staging" -> connectToStagingDB();
            case "dev"  -> connectToDevDB();
            default -> {
                System.err.println("Unknown env: " + env);
                System.exit(1);
            }
        }
    }
    // java App prod  → connects to production DB
    // java App       → defaults to dev (safe fallback)
```  </Card>

  <Card title="File processing CLI tool" icon="information">
    Real-world pattern for batch operations:
```java
    public class DataImporter {
        public static void main(String[] args) {
            if (args.length < 2) {
                System.out.println("Usage: java DataImporter <input.csv> <output.json> [--dry-run]");
                return;
            }
            
            String inputFile = args[0];
            String outputFile = args[1];
            boolean dryRun = Arrays.asList(args).contains("--dry-run");
            
            if (dryRun) {
                System.out.println("[DRY RUN] Would import " + inputFile + " → " + outputFile);
            } else {
                importData(inputFile, outputFile);
            }
        }
    }
    // java DataImporter data.csv result.json --dry-run
```
  </Card>
</CardGrid>

<Code lang="java" title="Advanced: Apache Commons CLI pattern" code={`// For complex CLI apps, use a library instead of manual parsing:
// Dependency: org.apache.commons:commons-cli

import org.apache.commons.cli.*;

public class AdvancedApp {
    public static void main(String[] args) {
        Options options = new Options();
        options.addOption("i", "input", true, "Input file path");
        options.addOption("o", "output", true, "Output file path");
        options.addOption("v", "verbose", false, "Enable verbose logging");
        
        CommandLineParser parser = new DefaultParser();
        try {
            CommandLine cmd = parser.parse(options, args);
            
            String input = cmd.getOptionValue("i", "default.txt");
            String output = cmd.getOptionValue("o", "result.json");
            boolean verbose = cmd.hasOption("v");
            
            process(input, output, verbose);
        } catch (ParseException e) {            System.err.println("Invalid arguments: " + e.getMessage());
            HelpFormatter formatter = new HelpFormatter();
            formatter.printHelp("java AdvancedApp", options);
        }
    }
}`} />


<Aside type="caution">
**Production Checklist** for CLI apps:1. ✅ Always validate `args.length` before indexing
2. ✅ Wrap parsing in try-catch for user-friendly errors
3. ✅ Print usage help when arguments are missing/invalid
4. ✅ Exit with non-zero code (`System.exit(1)`) on failure for script integration
5. ✅ Document expected arguments in `--help` output
</Aside>
