## 🔢 Varargs (Variable Arguments) in Java — Complete Guide

> Java 1.5+ **Varargs** allows a method to accept **zero or more arguments** of the same type — without passing an array explicitly.

---

## 📌 Syntax

<Code lang="java" title="Without vs. With Varargs" code={`// Without varargs — ugly, multiple overloads needed
void sum(int a) { }
void sum(int a, int b) { }
void sum(int a, int b, int c) { }  // 😩 never ending

// With varargs — one method handles all! ✅
void sum(int... nums) { }   // 0, 1, 2, 3... any number of ints!`} />

| Feature | Detail |
|---------|--------|
| **Symbol** | Three dots `...` (ellipsis) |
| **Type Meaning** | `int...` = "an array of ints" (`int[]`) |
| **Internal Representation** | Compiler converts to array automatically |

---

## 🔬 In-Depth — How Varargs Works Under the Hood

Varargs are **Syntactic Sugar**. The compiler converts them into an **Array**.

<Code lang="java" title="Varargs = Array internally" code={`void sum(int... nums) {
    System.out.println(nums.getClass().getName()); // [I  (int array!)
    System.out.println(nums.length);
    // 'nums' behaves exactly like int[] inside method
}`} />

### 💻 Calling Examples
<Code lang="java" title="All valid call patterns" code={`public static void sum(int... nums) {
    // Internally, 'nums' is treated as int[]
    System.out.println(nums.length); 
}

// Calling patterns:
sum();                    // ✅ nums = new int[0] (Empty array)
sum(1);                   // ✅ nums = new int[]{1}
sum(1, 2, 3);             // ✅ nums = new int[]{1, 2, 3}
sum(new int[]{1, 2});     // ✅ Array also works directly!`} />

### 🧠 Memory Diagram

```text
+---------------------+       +---------------------+
|      Stack          |       |        Heap         |
+---------------------+       +---------------------+
|  sum(1, 2, 3)       |       |  [int[] | 1, 2, 3]  |
|                     |       |                     |
|  [nums | @0x55]     |-------+--> (Array Object)   |
+---------------------+       +---------------------+
   Var-arg becomes a Reference to Array
```

<Aside type="note">
**Key Insight**: Inside the method, `nums` is a **reference variable** pointing to an array object on the Heap. All array operations (`.length`, indexing, iteration) work identically.
</Aside>

## ⚠️ Rules — Must Know

### Rule 1 — Varargs Must Be LAST Parameter

<Code lang="java" title="✅ Valid — varargs last" code={`void show(String name, int... scores) { }void show(int a, String b, double... vals) { }`} />

<Code lang="java" title="❌ Invalid — varargs not last" code={`void show(int... scores, String name) { }  // ❌ C.E: varargs must be last
void show(int... nums, String msg) { }      // ❌ C.E: same reason`} />

### Rule 2 — Only ONE Varargs Per Method

<Code lang="java" title="❌ Multiple varargs not allowed" code={`void show(int... a, int... b) { }  // ❌ C.E: only one varargs allowed per method`} />

### Rule 3 — Varargs vs Array Parameter: Duplicate Signature

<Code lang="java" title="❌ Cannot overload varargs and array" code={`// These TWO cause COMPILE ERROR — seen as same signature!
void sum(int[] nums) { }
void sum(int... nums) { }
// ❌ Duplicate method sum(int[]) in same class!

// Compiler treats int... and int[] as IDENTICAL signatures`} />

### Rule 4 — Mixing Varargs with General Parameters

<Code lang="java" title="✅ Valid combinations" code={`void methodOne(int a, int... b) { }           // ✅ valid
void methodOne(String s, int... x) { }        // ✅ valid
void methodOne(int a, String b, double... d) { } // ✅ valid`} />

### Rule 5 — Varargs Get Lowest Priority in Overloading

Varargs methods are chosen **only if no other method matches** — like a `default` case in `switch`.

<Code lang="java" title="Overloading priority demonstration" code={`class Test {
    public static void methodOne(int i) {
        System.out.println("general method");
    }
    public static void methodOne(int... i) {
        System.out.println("var-arg method");
    }
    public static void main(String[] args) {
        methodOne();          // var-arg method (no exact match)
        methodOne(10, 20);    // var-arg method (no exact match)
        methodOne(10);        // general method (exact match wins!)
    }
}`} />

### Rule 6 — Can Pass Array to Varargs Parameter

<Code lang="java" title="✅ Array argument accepted" code={`class Test {
    public static void methodOne(int... i) {
        System.out.println("var-arg method");
    }
    public static void main(String[] args) {
        methodOne(new int[]{10, 20, 30});  // ✅ var-arg method    }
}`} />

### Rule 7 — Cannot Declare Both `int[]` and `int...` in Same Class

<Code lang="java" title="❌ Duplicate method signature" code={`class Test {
    public void methodOne(int[] i) {}
    public void methodOne(int... i) {}  // ❌ C.E: method already defined
}`} />

# 4️⃣ Overloading with Var-Args (Interview Trap) 🚨

*This is the #1 Interview Question on this topic.*

### 🧠 Method Resolution Priority

When calling a method, Java chooses in this **strict order**:

1.  **Exact Match** (Fixed arguments) ← Highest priority
2.  **Widening** (`int` → `long`)
3.  **Autoboxing** (`int` → `Integer`)
4.  **Var-Arg** (Last priority) ← Lowest

### 💻 Example: Exact Match Wins

<Code lang="java" title="Fixed method beats varargs" code={`class Test {
    void show(int a) { System.out.println("Fixed"); }
    void show(int... a) { System.out.println("Var-Arg"); }
}

// Calling
new Test().show(10); 
// Output: "Fixed" 
// ✅ Exact match wins over Var-Arg`} />

### ⚠️ Ambiguity Error: Multiple Varargs

<Code lang="java" title="❌ Ambiguous overload" code={`class Test {
    void show(int... a) { System.out.println("Int"); }
    void show(long... a) { System.out.println("Long"); }
}

// Calling
new Test().show(10); 
// ❌ Compile Error: Reference to show is ambiguous
// Java doesn't know whether to box to int[] or long[]`} />

<Aside type="danger">**Pro Tip**: Avoid overloading methods that differ only by varargs type (`int...` vs `long...`). Use distinct method names instead to prevent ambiguity. </Aside>

# 📊 Single Dimensional Array vs Var-Arg Method

## 🔬 Declaration Difference

<Code lang="java" title="Syntax comparison" code={`// Single Dimensional Array parameter
void sum(int[] nums) { }

// Var-Arg Method  
void sum(int... nums) { }`} />

## 🔄 Calling Difference — The Biggest Difference

<Code lang="java" title="Array parameter — strict" code={`// Array parameter — caller MUST pass array explicitly
sum(new int[]{1, 2, 3});   // ✅ only way
sum(1, 2, 3);              // ❌ COMPILE ERROR: incompatible types`} />

<Code lang="java" title="Varargs — flexible" code={`// Var-Arg — caller can pass comma-separated values
sum(1, 2, 3);              // ✅
sum(new int[]{1, 2, 3});   // ✅ also works!
sum();                     // ✅ zero args allowed!`} />

## 🔬 Advanced: Array of Varargs (2D Case)

<Code lang="java" title="int[]... creates 2D array internally" code={`// Note: 
// 1. methodOne(int... x) 
//    → x becomes 1D array (int[] x)
//
// 2. methodOne(int[]... x) 
//    → x becomes 2D array (int[][] x)
//    → We pass group of 1D int[] arrays

class Test {
    public static void methodOne(int[]... x) {
        for(int[] a : x) {
            System.out.println(a[0]);
        }
    }
    public static void main(String[] args) {
        int[] l = {10, 20, 30};
        int[] m = {40, 50};
        methodOne(l, m);  // Pass multiple 1D arrays
    }
}
// Output:
// 10
// 40`} />
<Aside type="note">
**Why Case 2 replacement is invalid**: `methodOne(int... i)` allows flexible calling (`methodOne(1,2,3)`). Replacing with `methodOne(int[] i)` forces caller to always wrap values in `new int[]{...}`, losing the varargs convenience.
</Aside>

## 🎯 Interview Cheat Sheet

<CardGrid>
  <Card title="Q: Can you pass array to vararg method?" icon="approve-check">
    **YES ✅** — `int[]` is directly accepted by `int...` parameter.
```java
    void sum(int... nums) { }
    int[] arr = {1, 2, 3};
    sum(arr);  // ✅ Works perfectly
```
  </Card>

  <Card title="Q: Can you pass comma-separated values to array param?" icon="error">
    **NO ❌** — must wrap in `new int[]{...}` explicitly.
```java
    void sum(int[] nums) { }
    sum(1, 2, 3);  // ❌ C.E: incompatible types
    sum(new int[]{1, 2, 3});  // ✅ Only valid way
```
  </Card>

  <Card title="Q: Which is better — array param or varargs?" icon="information">
    **Varargs** — more flexible for callers.  
    Use array param only when:
    - Caller always has array ready
    - You need to accept `null` explicitly as a distinct case
    - Performance-critical code (minor: avoids array creation overhead)
  </Card>

  <Card title="Q: What does varargs do with zero arguments?" icon="backspace">
    Creates **empty array** — `int[]` of length 0.
```java
    void sum(int... nums) { }
    sum();  // nums = new int[0], NOT null!
    
    // Contrast with array param:
    void sumArr(int[] nums) { }
    sumArr();        // ❌ C.E: must pass argument
    sumArr(null);    // ✅ nums = null (distinct from empty array)
    sumArr(new int[0]); // ✅ nums = empty array
```
  </Card>
  <Card title="Q: What is varargs internally?" icon="information">
    **It's an array** — compiler converts arguments into array automatically.
```java
    void show(int... nums) {
        // Inside: nums IS an int[] reference
        System.out.println(nums instanceof int[]); // true
    }
```
  </Card>

  <Card title="Q: Can varargs be null?" icon="error">
    **YES** — but be careful!
```java
    void sum(int... nums) {
        // nums can be null if caller passes null explicitly:
        // sum(null);
        
        // ⚠️ Danger: accessing null array throws NPE
        // System.out.println(nums[0]); // 💥 if nums == null
        
        // ✅ Safe pattern:
        if (nums == null) return;
        for (int n : nums) { /* safe */ }
    }
```
  </Card>

  <Card title="Q: Object... trap with primitive arrays" icon="error">
```java
    void show(Object... args) { 
        System.out.println(args.length); 
    }
    
    show(new int[]{1, 2, 3});  // → Output: 1 !! 😱
    // Entire int[] is treated as ONE Object argument!
    // Because int[] is an Object, not an int.
    
    // ✅ Fix: Use Integer... for boxed types
    void show(Integer... args) { 
        show(1, 2, 3);  // → Output: 3 ✅
    }
```
  </Card>
</CardGrid>

## 🧩 DSA Angle

<CardGrid>  <Card title="Pattern: Utility Methods" icon="rocket">
    Varargs excel for helper functions in DSA:
```java
    // Find max of any number of values
    int max(int... nums) {
        int maxVal = Integer.MIN_VALUE;
        for (int n : nums) maxVal = Math.max(maxVal, n);
        return maxVal;
    }
    max(3, 1, 4, 1, 5, 9);  // ✅ Clean call
```
  </Card>
  
  <Card title="Pattern: Test Case Helpers" icon="shield">
    Quickly test functions with variable inputs:
```java
    void test(int expected, int... inputs) {
        int result = solve(inputs);
        System.out.println(result == expected ? "PASS" : "FAIL");
    }
    test(6, 1, 2, 3);        // sum = 6
    test(15, 1, 2, 3, 4, 5); // sum = 15
```
  </Card>

  <Card title="Pattern: When to Use Array Param" icon="information">
    Use array parameter when input is already an array:
```java
    // Array param — when you already have array from input
    int[] arr = {1, 2, 3};
    solve(arr);             // natural fit, no wrapping needed
    
    // Varargs — better for ad-hoc value lists
    int result = compute(1, 2, 3);  // cleaner than new int[]{1,2,3}
```
  </Card>
</CardGrid>

## 🏢 Real-Life SDE Usage

<Code lang="java" title="Built-in Java varargs examples" code={`// 1. Standard library uses varargs extensively:
System.out.printf("%s is %d years old", "Alice", 25); 
// String.format, MessageFormat, etc.

Arrays.asList(1, 2, 3, 4);     // Collections utility
List.of("a", "b", "c");        // Java 9+ factory method

// 2. Logging utilityvoid log(String level, String... messages) {
    for (String msg : messages) {
        System.out.println("[" + level + "] " + msg);
    }
}
log("INFO", "Server started", "Port: 8080", "DB connected");

// 3. Database query builder
String buildQuery(String table, String... columns) {
    return "SELECT " + String.join(", ", columns)
           + " FROM " + table;
}
buildQuery("users", "id", "name", "email");
// → SELECT id, name, email FROM users ✅

// 4. Validation helper
boolean allPositive(int... numbers) {
    for (int n : numbers) if (n <= 0) return false;
    return true;
}
allPositive(1, 2, 3);   // true
allPositive(1, -1, 3);  // false`} />

<Aside type="tip">
**Best Practice**: Use varargs for methods where the number of arguments is naturally variable (logging, formatting, validation). Use explicit array parameters when the input is inherently a collection from another source (file I/O, API response, database result).
</Aside>

## 🔑 Quick Reference Summary

| Rule | Description | Example |
|------|-------------|---------|
| **Position** | Varargs must be last parameter | `void m(String s, int... nums)` ✅ |
| **Count** | Only one varargs per method | `void m(int... a, int... b)` ❌ |
| **Signature** | `int...` and `int[]` are identical to compiler | Cannot overload both |
| **Priority** | Varargs chosen last in overload resolution | Exact match > Widening > Autoboxing > Varargs |
| **Zero Args** | Creates empty array, not `null` | `sum()` → `nums = new int[0]` |
| **Null Arg** | Can pass `null` explicitly | `sum(null)` → `nums = null` |
| **Array Arg** | Can pass array directly | `sum(new int[]{1,2})` ✅ |

<Aside type="caution">
**Final Warning**: Never write `if (nums != null)` for varargs unless you explicitly allow `null` as a valid argument. By default, varargs create an empty array for zero arguments — `null` is a distinct, intentional case.
</Aside>
