## 🎭 Abstraction in Java — Complete Guide

> **Abstraction** is the OOP principle of **hiding complex implementation details** and exposing only essential features to the user. It focuses on **what** an object does, not **how** it does it.

```java
// ❌ Without abstraction — client depends on concrete implementation
class PaymentProcessor {
    public void processCreditCard(String cardNumber, double amount) {
        // Direct credit card logic — hard to change, test, or extend
        validateCard(cardNumber);
        chargeCard(cardNumber, amount);
        logTransaction("CREDIT", amount);
    }
}

// ✅ With abstraction — client depends on contract
interface PaymentMethod {
    void process(double amount);  // WHAT to do
}

class CreditCardPayment implements PaymentMethod {
    @Override
    public void process(double amount) {
        // HOW to do it — hidden from client
        validateCard(cardNumber);
        chargeCard(cardNumber, amount);
        logTransaction("CREDIT", amount);
    }
}

// Client code — unchanged when adding new payment methods
class CheckoutService {
    private final PaymentMethod payment;  // Depend on abstraction!
    
    CheckoutService(PaymentMethod payment) {
        this.payment = payment;  // Inject concrete implementation
    }
    
    void checkout(double amount) {
        payment.process(amount);  // Polymorphic call — works with ANY PaymentMethod
    }
}
```

<Aside type="tip">
**Key Insight**: Abstraction is not about hiding code — it's about **managing complexity** and **enabling flexibility**.  
By programming to abstractions (interfaces/abstract classes), you:
- ✅ Decouple components — change implementation without breaking clients
- ✅ Enable testing — mock abstractions for unit tests
- ✅ Support extensibility — add new implementations without modifying existing code
- ✅ Clarify intent — contracts document expected behavior
</Aside>

## 🗂️ Two Mechanisms for Abstraction in Java

> Java provides **abstract classes** and **interfaces** to achieve abstraction — each with distinct use cases.

### 🔑 Abstract Class vs Interface — Quick Comparison

<table>
  <thead>
    <tr>
      <th>Feature</th>
      <th>Abstract Class</th>
      <th>Interface</th>
    </tr>
  </thead>
  <tbody>
    <tr><td><strong>Relationship</strong></td><td>IS-A (type hierarchy)</td><td>CAN-DO (capability)</td></tr>

### 💻 Code Comparison — Same Goal, Different Approaches

<Code lang="java" title="Abstract class — shared state + partial implementation" code={`// Abstract base for related animals
abstract class Animal {
    // Shared state
    protected String name;
    protected int age;
    
    // Constructor — initialize shared state for child
    Animal(String name, int age) {
        this.name = name;
        this.age = age;
    }
    
    // Concrete method — shared behavior
    public void breathe() {
        System.out.println(name + " is breathing...");
    }
    
    // Abstract method — each subclass provides own implementation
    public abstract void makeSound();
    
    // Protected hook — extension point for subclasses
    protected void onWakeUp() {
        // Default behavior — can be overridden
        System.out.println(name + " wakes up");
    }
}

// Concrete subclass — completes the contract
class Dog extends Animal {
    Dog(String name, int age) {
        super(name, age);  // Initialize shared state
    }
    
    @Override
    public void makeSound() {
        System.out.println(name + " says: Woof!");
    }
    
    @Override
    protected void onWakeUp() {
        super.onWakeUp();  // Reuse base behavior
        System.out.println(name + " wags tail");  // Add dog-specific behavior
    }
}

// Usage — polymorphism via abstract type
Animal pet = new Dog("Buddy", 3);
pet.breathe();      // "Buddy is breathing..." (shared implementation)
pet.makeSound();    // "Buddy says: Woof!" (polymorphic call)
`} />

<Code lang="java" title="Interface — capability for unrelated classes" code={`// Capability: things that can fly
interface Flyable {
    // Abstract method — contract only
    void fly();
    
    // Default method — optional override (Java 8+)
    default void land() {
        System.out.println("Landing safely...");
    }
    
    // Static utility — called via interface name
    static double calculateLift(double weight, double wingArea) {
        return 0.5 * 1.225 * wingArea * Math.pow(10, 2);  // Simplified physics
    }
}

// Unrelated classes implement same capability
class Bird implements Flyable {
    @Override
    public void fly() {
        System.out.println("Flapping wings...");
    }
    // land() uses default implementation ✅
}

class Airplane implements Flyable {
    @Override
    public void fly() {
        System.out.println("Engines at full thrust...");
    }
    
    @Override
    public void land() {
        System.out.println("Deploying landing gear...");  // Custom override
    }
}

class Superman implements Flyable {
    @Override
    public void fly() {
        System.out.println("Up, up, and away!");
    }
    // land() uses default implementation ✅
}

// Usage — polymorphism via interface type
List<Flyable> flyers = List.of(new Bird(), new Airplane(), new Superman());
for (Flyable flyer : flyers) {
    flyer.fly();   // Polymorphic call — each implements differently
    flyer.land();  // Default or overridden — works for all
}

// Static method — called via interface, not instance
double lift = Flyable.calculateLift(70, 2.5);
`} />

<Aside type="note">
**Design Heuristic**:  
- Use **abstract class** when classes are **related** (IS-A) and share **state or common implementation**.  
- Use **interface** when classes are **unrelated** but need the **same capability** (CAN-DO).  
- You can **combine both**: `class Helicopter extends Aircraft implements Flyable, Hoverable`
</Aside>

## 🔧 Abstract Classes — Deep Dive

### 🎯 When to Use Abstract Classes

<CardGrid>
  <Card title="Use Abstract Class When…" icon="rocket">
    ```text
    ✅ You need to share code (fields or concrete methods) among subclasses
       • Abstract base provides common implementation, subclasses customize
    
    ✅ You need to declare non-public (protected/private) members
       • Interfaces only support public methods (pre-Java 9)
    
    ✅ You need to declare instance fields that aren't static or final
       • Interfaces only allow public static final constants
    
    ✅ You anticipate adding new methods later without breaking subclasses
       • Add concrete methods to abstract class — subclasses inherit automatically
    ```
  </Card>
  
  <Card title="Abstract Class Structure Pattern" icon="shield">
    ```java
    public abstract class BaseRepository<T, ID> {
        // Shared state — protected for subclass access
        protected final DataSource dataSource;
        protected final Class<T> entityType;
        
        // Constructor — initialize shared dependencies
        protected BaseRepository(DataSource dataSource, Class<T> entityType) {
            this.dataSource = Objects.requireNonNull(dataSource);
            this.entityType = Objects.requireNonNull(entityType);
        }
        
        // Concrete method — reusable implementation
        public Optional<T> findById(ID id) {
            String sql = buildSelectQuery();  // Abstract hook
            return dataSource.query(sql, id, this::mapRowToEntity);
        }
        
        // Abstract hook — subclasses provide entity-specific logic
        protected abstract String buildSelectQuery();
        protected abstract T mapRowToEntity(ResultSet rs) throws SQLException;
        
        // Protected helper — extension point, not public API
        protected void validateEntity(T entity) {
            // Default validation — can be overridden
            if (entity == null) throw new IllegalArgumentException();
        }
    }
    
    // Concrete subclass — completes the contract
    public class UserRepository extends BaseRepository<User, Long> {
        public UserRepository(DataSource dataSource) {
            super(dataSource, User.class);
        }
        
        @Override
        protected String buildSelectQuery() {
            return "SELECT * FROM users WHERE id = ?";  // Entity-specific SQL
        }
        
        @Override
        protected User mapRowToEntity(ResultSet rs) throws SQLException {
            return new User(
                rs.getLong("id"),
                rs.getString("email"),
                rs.getString("name")
            );
        }
        
        // Add entity-specific methods
        public Optional<User> findByEmail(String email) {
            // Custom query — not in base class
            String sql = "SELECT * FROM users WHERE email = ?";
            return dataSource.query(sql, email, this::mapRowToEntity);
        }
    }
    ```

### ⚠️ Abstract Class Rules & Common Pitfalls

<CardGrid>
  <Card title="Rule: Cannot Instantiate Abstract Class" icon="error">
    ```java
    abstract class Shape {
        abstract double area();
    }
    
    // ❌ Compile Error: Shape is abstract; cannot be instantiated
    Shape s = new Shape();
    
    // ✅ Correct: Instantiate concrete subclass
    Shape circle = new Circle(5);  // Circle extends Shape
    ```
  </Card>
  
  <Card title="Rule: Subclass Must Implement All Abstract Methods (or be abstract)" icon="approve-check">
    ```java
    abstract class Animal {
        abstract void makeSound();
        abstract void move();
    }
    
    // ❌ Compile Error: Dog is not abstract and does not override abstract methods
    class Dog extends Animal {
        @Override
        void makeSound() { System.out.println("Woof"); }
        // move() not implemented → C.E
    }
    
    // ✅ Option 1: Implement all abstract methods
    class Dog extends Animal {
        @Override
        void makeSound() { System.out.println("Woof"); }
        @Override
        void move() { System.out.println("Running"); }
    }
    
    // ✅ Option 2: Declare subclass abstract (defer to its subclass)
    abstract class Mammal extends Animal {
        @Override
        void makeSound() { System.out.println("Mammal sound"); }
        // move() still abstract — concrete subclass must implement
    }
    ```
  </Card>

  <Card title="Pitfall: Overusing Abstract Classes for Simple Contracts" icon="caution">
    ```java
    // ❌ Over-engineered: Simple contract doesn't need abstract class
    abstract class Logger {
        public abstract void log(String message);
        // No shared state, no concrete methods — just a contract
    }
    
    // ✅ Better: Use interface for pure contracts
    interface Logger {
        void log(String message);  // Implicitly public abstract
    }
    
    // Implementations remain the same:
    class ConsoleLogger implements Logger {
        @Override
        public void log(String message) {
            System.out.println(message);
        }
    }
    ```
  </Card>
</CardGrid>

## 🔌 Interfaces — Deep Dive

### 🎯 When to Use Interfaces

<CardGrid>
  <Card title="Use Interface When…" icon="rocket">
    ```text
    ✅ Unrelated classes need the same capability
       • Bird, Airplane, Superman → all implement Flyable
    
    ✅ You want to support multiple inheritance of type
       • class SmartWatch implements Displayable, Connectable, Trackable
    
    ✅ You're defining a public API/contract for others to implement
       • Java Collections: List, Set, Map interfaces
    
    ✅ You need to add methods later without breaking existing implementations
       • Use default methods (Java 8+) for backward-compatible evolution
    ```
  </Card>
  
  <Card title="Interface Evolution with Default Methods" icon="shield">
    ```java
    // Java 7: Interface with one method
    interface List<E> {
        void add(E e);
        E get(int index);
        // All implementing classes work fine ✅
    }
    
    // Java 8: Add new method with default implementation → no breakage!
    interface List<E> {
        void add(E e);
        E get(int index);
        
        // New method with default body — existing classes inherit it ✅
        default void forEach(Consumer<? super E> action) {
            for (E e : this) {  // 'this' refers to the implementing List
                action.accept(e);
            }
        }
    }
    
    // Existing ArrayList implementation — no changes needed!
    class ArrayList<E> implements List<E> {
        // Existing methods unchanged
        // Inherits forEach() default implementation automatically ✅
        
        // Optional: Override for optimization
        @Override
        public void forEach(Consumer<? super E> action) {
            // More efficient array-based iteration
            for (int i = 0; i < size; i++) {
                action.accept(elementData[i]);
            }
        }
    }
    ```

    ```java
    interface A {
        default void show() { System.out.println("A"); }
    }
    interface B {
        default void show() { System.out.println("B"); }
    }

        @Override
        public void show() {

            A.super.show();  // "A"

        }
    }
    ```

## 🎭 Abstract Methods — The Heart of Abstraction

> Abstract methods define the **contract** — subclasses must provide the **implementation**.

### 🔑 Abstract Method Rules

<CardGrid>
  <Card title="Implicit Modifiers — All Declarations Are Equal" icon="approve-check">
    ```java
    interface Shape {
        // All four declarations are IDENTICAL:
        double area();                        // implicit: public abstract
        public double perimeter();            // explicit public
        abstract double circumference();      // explicit abstract
        public abstract double diameter();    // explicit both
    }
    
    // Compiler treats all as: public abstract double methodName();
    ```
  </Card>
  
  <Card title="Illegal Modifier Combinations" icon="error">
    ```java
    abstract class Task {
        // ❌ abstract + final: contradiction — abstract requires override, final prevents it
        // abstract final void execute();  // C.E
        
        // ❌ abstract + private: child can't see or override private method
        // abstract private void run();  // C.E
        
        // ❌ abstract + static: static resolved at compile time, abstract needs runtime polymorphism
        // abstract static void init();  // C.E
        
        // ❌ abstract + native: both imply "no Java body" → redundant/contradictory
        // abstract native void process();  // C.E
        
        // ✅ Valid: abstract alone, or with public/protected
        public abstract void execute();
        protected abstract void validate();
    }
    ```
  </Card>

  <Card title="Implementation Must Strengthen or Match Access" icon="caution">
    ```java
    abstract class Service {
        protected abstract void process();  // protected abstract
    }
    
    class ConcreteService extends Service {
        // ❌ C.E: process() in ConcreteService cannot implement process() in Service;
        // attempting to assign weaker access privileges ('package-private'); was 'protected'
        // void process() { }  // Weaker than protected
        
        // ✅ Correct: match or strengthen access
        @Override
        protected void process() { }  // Match protected
        // OR
        @Override
        public void process() { }     // Strengthen to public (allowed)
    }
    ```
  </Card>
</CardGrid>

### 💻 Abstract Method Patterns

<Code lang="java" title="Template Method Pattern — abstract hooks for customization" code={`// Abstract base defines algorithm skeleton
public abstract class DataProcessor {
    
    // Template method — final to prevent overriding skeleton
    public final void process(String input) {
        validate(input);           // Concrete step
        String cleaned = clean(input);  // Abstract hook
        String transformed = transform(cleaned);  // Abstract hook
        save(transformed);         // Concrete step
        log("Processed: " + transformed);  // Concrete step
    }
    
    // Concrete method — shared implementation
    protected void validate(String input) {
        if (input == null || input.isEmpty()) {
            throw new IllegalArgumentException("Input required");
        }
    }
      // Abstract hooks — subclasses provide entity-specific logic
    protected abstract String clean(String input);
    protected abstract String transform(String cleaned);
    
    // Concrete method — reusable
    protected void save(String data) {
        // Default save logic — can be overridden if needed
        System.out.println("Saving: " + data);
    }
    
    protected void log(String message) {
        System.out.println("[LOG] " + message);
    }
}

// Concrete subclass — implements hooks
public class CsvProcessor extends DataProcessor {
    @Override
    protected String clean(String input) {
        return input.trim().replaceAll("\\s+", " ");
    }
    
    @Override
    protected String transform(String cleaned) {
        return cleaned.toUpperCase();  // CSV-specific transformation
    }
    
    // Optional: override concrete method for custom behavior
    @Override
    protected void save(String data) {
        // CSV-specific save logic
        try (FileWriter writer = new FileWriter("output.csv")) {
            writer.write(data);
        } catch (IOException e) {
            throw new RuntimeException(e);
        }
    }
}

// Usage — same interface, different behavior
DataProcessor csv = new CsvProcessor();
csv.process("  hello   world  ");  // Logs: "Saving: HELLO WORLD"
`} />

<Aside type="tip">
**Template Method Benefits**:
- ✅ Enforces consistent algorithm structure across subclasses
- ✅ Allows customization at defined extension points (abstract hooks)
- ✅ Prevents accidental modification of core algorithm (`final` template method)
- ✅ Promotes code reuse — concrete methods shared by all subclasses
</Aside>

## 🧩 Abstraction in DSA — Practical Patterns

<CardGrid>
  <Card title="Pattern: Abstract Base for Graph Algorithms" icon="rocket">
    ```java
    // Abstract base — defines graph interface
    public abstract class Graph {
        protected final int vertices;
        
        protected Graph(int vertices) {
            this.vertices = vertices;
        }
        
        // Abstract methods — concrete graph types implement
        public abstract void addEdge(int src, int dest);
        public abstract List<Integer> getNeighbors(int vertex);
        
        // Concrete algorithm — works for ANY graph implementation
        public List<Integer> bfs(int start) {
            boolean[] visited = new boolean[vertices];
            Queue<Integer> queue = new LinkedList<>();
            List<Integer> result = new ArrayList<>();
            
            queue.offer(start);
            visited[start] = true;
            
            while (!queue.isEmpty()) {
                int vertex = queue.poll();
                result.add(vertex);
                
                for (int neighbor : getNeighbors(vertex)) {  // Abstract hook
                    if (!visited[neighbor]) {
                        visited[neighbor] = true;
                        queue.offer(neighbor);
                    }
                }
            }
            return result;
        }
    }
    
    // Concrete adjacency list implementation
    public class AdjacencyListGraph extends Graph {
        private final List<List<Integer>> adj;
        
        public AdjacencyListGraph(int vertices) {
            super(vertices);
            adj = new ArrayList<>(vertices);
            for (int i = 0; i < vertices; i++) adj.add(new ArrayList<>());
        }
        
        @Override
        public void addEdge(int src, int dest) {
            adj.get(src).add(dest);
            adj.get(dest).add(src);  // Undirected
        }
        
        @Override
        public List<Integer> getNeighbors(int vertex) {
            return Collections.unmodifiableList(adj.get(vertex));
        }
    }
    
    // Usage — algorithm works with any Graph implementation
    Graph graph = new AdjacencyListGraph(5);
    graph.addEdge(0, 1);
    graph.addEdge(0, 2);
    List<Integer> traversal = graph.bfs(0);  // Polymorphic call ✅
    ```

    ```java
    // Iterator interface — abstracts traversal logic
    public interface Iterator<T> {
        boolean hasNext();
        T next();
        default void remove() { throw new UnsupportedOperationException(); }
    }
    
    // Iterable interface — enables for-each loop
    public interface Iterable<T> {
        Iterator<T> iterator();
    }
    
    // Custom collection implements Iterable
    public class MyList<T> implements Iterable<T> {
        private T[] elements;
        private int size;
        
        @Override
        public Iterator<T> iterator() {
            return new Iterator<T>() {
                private int index = 0;
                @Override
                public boolean hasNext() { return index < size; }
                @Override
                public T next() { 
                    if (!hasNext()) throw new NoSuchElementException();
                    return elements[index++]; 
                }
            };
        }
    }
    
    // Usage — works with for-each loop (syntactic sugar for iterator)
    MyList<String> list = new MyList<>();
    for (String item : list) {  // Compiler: list.iterator().next()...
        System.out.println(item);
    }
    
    // Algorithm — works with ANY Iterable, not just ArrayList
    public static <T> void printAll(Iterable<T> items) {
        for (T item : items) {  // Polymorphic iteration ✅
            System.out.println(item);
        }
    }
    ```

## 🎯 Interview Cheat Sheet

<CardGrid>
  <Card title="Q: What is the difference between abstraction and encapsulation?" icon="information">
    **Abstraction**: Hiding implementation details, exposing only essential features (focus on **what**).  
    **Encapsulation**: Bundling data + methods, restricting direct access to state (focus on **how to protect**).  
    ✅ Abstraction is about design-level simplification; encapsulation is about implementation-level protection.
  </Card>
  
  <Card title="Q: When would you choose an abstract class over an interface?" icon="rocket">
    Choose **abstract class** when:
    - Classes are **related** (IS-A hierarchy) and share **state or common implementation**
    - You need **non-public members** (protected/private methods or fields)
    - You need **constructors** to initialize shared state for subclasses
    - You anticipate adding **concrete methods** later without breaking subclasses
    
    Choose **interface** when:
    - Classes are **unrelated** but need the **same capability** (CAN-DO)
    - You need **multiple inheritance** of type
    - You're defining a **public API contract** for external implementers
  </Card>

  <Card title="Q: Can an abstract class have a constructor?" icon="approve-check">
    **YES ✅** — abstract classes can (and often should) have constructors.  
    Purpose: Initialize **instance fields** that subclasses inherit.  
    Called via `super()` from subclass constructor — initializes shared state before subclass-specific logic.
  </Card>

  <Card title="Q: Can an interface have a constructor?" icon="error">
    **NO ❌** — interfaces cannot be instantiated and have no instance fields to initialize.  
    Interface variables are `public static final` — initialized at declaration or in static blocks, not per-instance.
  </Card>

  <Card title="Q: What happens if a class implements an interface but doesn't implement all methods?" icon="error">
    **Compile Error** — unless the class is declared `abstract`.  
    - Concrete class: Must implement **all** abstract methods from interfaces + abstract superclasses  
    - Abstract class: Can leave some methods unimplemented — concrete subclasses must complete the contract
  </Card>

  <Card title="Q: How do default methods solve the interface evolution problem?" icon="shield">
    **Problem**: Adding a new abstract method to an interface breaks all existing implementations.  
    **Solution**: Default methods provide a **default implementation** — existing classes inherit it automatically.  
    Example: Java 8 added `forEach()` to `Iterable` with a default implementation — no existing `Collection` implementations broke.
  </Card>

  <Card title="Q: Can you override a default method in a subclass?" icon="approve-check">
    **YES ✅** — default methods are virtual — subclasses can override them.  
    To call the interface's default implementation from within an override:  
    ```java
    @Override
    public void show() {
        MyInterface.super.show();  // Explicitly invoke default implementation
        // Add subclass-specific behavior
    }
    ```
  </Card>

  <Card title="Q: What is the 'open/closed principle' and how does abstraction help?" icon="rocket">
    **Open/Closed Principle**: Software entities should be **open for extension** but **closed for modification**.  
    **Abstraction enables this**:  
    - Define stable abstractions (interfaces/abstract classes) — "closed" to change  
    - Add new implementations (concrete classes) — "open" for extension  
    - Client code depends on abstractions — unchanged when adding new implementations ✅
  </Card>
</CardGrid>

## 🔑 Quick Reference Summary

### Abstraction Decision Guide

```text
Do you need to...?

┌─────────────────────────────────────┐
│ Share code (fields/methods) between related classes? │
│ ├─ Yes → Use abstract class ✅      │
│ └─ No  → Go to next question        │
├─────────────────────────────────────┤
│ Support multiple inheritance of type?│
│ ├─ Yes → Use interface ✅           │
│ └─ No  → Go to next question        │
│                                     │
│ Define a public API contract?       │
│ ├─ Yes → Use interface ✅           │
│ └─ No  → Consider concrete class   │
└─────────────────────────────────────┘
```

### Abstract Class vs Interface — When to Use

| Scenario | Abstract Class | Interface |
|----------|---------------|-----------|
| Related classes sharing state | ✅ Yes | ❌ No |
| Unrelated classes needing same capability | ❌ No | ✅ Yes |
| Need protected/private members | ✅ Yes | ❌ No (pre-Java 9) |
| Need constructors for initialization | ✅ Yes | ❌ No |
| Need multiple inheritance of type | ❌ No | ✅ Yes |
| Adding methods later without breaking impls | ✅ Add concrete methods | ✅ Add default methods |
| Defining public API for external use | ⚠️ Possible | ✅ Preferred |

### Common Pitfalls & Fixes

| Pitfall | Risk | Fix |
|---------|------|-----|
| Using abstract class for pure contract | Over-engineering, limits multiple inheritance | Use interface instead |
| Forgetting to implement all interface methods | Compile error | Implement all or declare class abstract |
| Default method conflict in multiple interfaces | Compile error | Override + use `InterfaceName.super.method()` |
| Exposing mutable state in abstract class | Subclass can break invariants | Use protected access + document invariants |
| Overusing abstraction for simple cases | Unnecessary complexity | Start concrete — abstract only when needed |

<Aside type="caution">
**Final Checklist for Abstraction Design**:
1. ✅ **Start concrete** — abstract only when you have multiple implementations or need to enforce a contract
2. ✅ **Program to interfaces** — use abstractions in public APIs, dependencies, and method signatures
3. ✅ **Document contracts clearly** — JavaDoc for abstract methods and interfaces
4. ✅ **Keep abstractions focused** — single responsibility per interface/abstract class
5. ✅ **Use default methods wisely** — for backward-compatible interface evolution
6. ✅ **Test via abstractions** — mock interfaces for isolated unit tests
7. ✅ **Avoid premature abstraction** — don't abstract until you have at least two concrete implementations or a clear need
</Aside>
