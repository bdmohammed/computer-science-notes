# 🎭 Polymorphism in Java — Complete Guide

> **Polymorphism** means **one name, many forms** — same method behaves differently based on context.

---

## 📌 Core Idea

```
Poly  = Many
Morph = Forms

Example 1: We' can use same abs() method for int type, long type, float type etc.
1. abs(int)
2. abs(long)
3. abs(float)

Example 2: 
We can use the parent reference to hold any child objects. We can use the same List reference to hold ArrayList object, LinkedList object, Vector object, or Stack object.
Example: 
1. List l=new ArrayList();
2. List l=new LinkedList();
3. List l=new Vector();
4. List l=new Stack();

Same method name → different behavior
depending on WHO calls it or WHAT arguments passed

Real World:
Person acts differently in different roles:
Same "Rahul" → Son at home
              → Employee at office
              → Friend with friends
Same entity, different behavior = Polymorphism!
```

## 1️⃣ Compile-Time Polymorphism — Method Overloading

> **Same method name, different parameters** — compiler decides which to call.

```java
class Calculator {
    // Same name — different parameter types/count
    int add(int a, int b) {
        return a + b;
    }
    double add(double a, double b) {
        return a + b;
    }
    int add(int a, int b, int c) {
        return a + b + c;
    }
    String add(String a, String b) {
        return a + b;
    }
}

Calculator c = new Calculator();
c.add(1, 2);          // int version    ← compiler picks at compile time
c.add(1.5, 2.5);      // double version
c.add(1, 2, 3);       // 3-arg version
c.add("Hi", " Java"); // String version
```

```
Overloading is resolved at COMPILE TIME
based on:
├── Number of parameters
├── Type of parameters
└── Order of parameters

NOT based on:
├── Return type alone ❌
└── Access modifier   ❌
```

### Overloading — What Counts & What Doesn't

```java
// ✅ Valid overloads
void show(int a) { }
void show(double a) { }       // different type
void show(int a, int b) { }   // different count
void show(String a, int b) { }
void show(int a, String b) { } // different order

// ❌ NOT valid overloads — compile error!
int  show(int a) { return 1; }
void show(int a) { }           // return type alone ❌ COMPILE ERROR!
```

### ⚠️ Overloading with Type Promotion
```java
class Demo {
    void show(int a)    { System.out.println("int");    }
    void show(long a)   { System.out.println("long");   }
    void show(double a) { System.out.println("double"); }
}

Demo d = new Demo();
d.show(10);    // int    ← exact match first!
d.show(10L);   // long
d.show(10.0);  // double
d.show('A');   // int    ← char promoted to int! (65)

// What if only long exists?
class Demo2 {
    void show(long a) { System.out.println("long"); }
}
new Demo2().show(10); // long ← int promoted to long ✅
```

```
Auto-promotion order (widening):
byte → short → int → long → float → double
char → int → long → float → double

If exact match not found → widens automatically!
```

## 2️⃣ Runtime Polymorphism — Method Overriding

> **Same method, different implementation** — JVM decides which to call at runtime.

```java
class Animal {
    void sound() {
        System.out.println("Some sound");
    }
}

class Dog extends Animal {
    @Override
    void sound() { System.out.println("Woof!"); }
}

class Cat extends Animal {
    @Override
    void sound() { System.out.println("Meow!"); }
}

class Duck extends Animal {
    @Override
    void sound() { System.out.println("Quack!"); }
}

// 🔥 Runtime Polymorphism in action!
Animal[] animals = {new Dog(), new Cat(), new Duck()};

for (Animal a : animals) {
    a.sound(); // SAME call — DIFFERENT result each time!
}
// Woof!
// Meow!
// Quack!
// JVM decides at runtime based on actual object type ✅
```

---

## 🔬 In-Depth — How Runtime Polymorphism Works

```java
Animal a = new Dog();
a.sound(); // How does JVM know to call Dog's sound()?
```

```
Step 1: Object created on Heap
┌─────────────────────────────────────┐
│           Dog Object                │
│  vtable pointer ────────────────────┼──▶ Dog's vtable
│  [Animal part]                      │    ┌──────────────────────┐
│  [Dog part]                         │    │ sound() → Dog.sound  │
└─────────────────────────────────────┘    │ eat()   → Animal.eat │
                                           └──────────────────────┘

Step 2: a.sound() called
├── JVM looks at ACTUAL object (Dog) not reference type (Animal)
├── Finds Dog's vtable
├── Looks up sound() → Dog.sound()
└── Executes Dog.sound() → "Woof!" ✅

This lookup happens at RUNTIME → Runtime Polymorphism!
```

```
Compile time thinks:  Animal reference → calls Animal.sound()
Runtime actually:     Dog object      → calls Dog.sound()
JVM always goes by ACTUAL object, not reference type!
```

## 🔄 Dynamic Dispatch — The Engine of Runtime Polymorphism

```java
class Shape {
    double area() { return 0; }
}

class Circle extends Shape {
    double radius;
    Circle(double r) { this.radius = r; }

    @Override
    double area() { return Math.PI * radius * radius; }
}

class Rectangle extends Shape {
    double w, h;
    Rectangle(double w, double h) { this.w = w; this.h = h; }

    @Override
    double area() { return w * h; }
}

class Triangle extends Shape {
    double base, height;
    Triangle(double b, double h) { this.base = b; this.height = h; }

    @Override
    double area() { return 0.5 * base * height; }
}

// Dynamic Dispatch — same reference, different dispatch!
Shape s;

s = new Circle(5);
System.out.println(s.area()); // 78.53 ← Circle's area()

s = new Rectangle(4, 6);
System.out.println(s.area()); // 24.0 ← Rectangle's area()

s = new Triangle(3, 8);
System.out.println(s.area()); // 12.0 ← Triangle's area()

// s.area() is the SAME call every time
// but dispatched to DIFFERENT methods at runtime!
```

## 🆚 Overloading vs Overriding — Full Comparison

```
┌──────────────────┬─────────────────────┬──────────────────────┐
│    Feature       │    Overloading      │     Overriding       │
├──────────────────┼─────────────────────┼──────────────────────┤
│ Type             │ Compile-time        │ Runtime              │
│ Class            │ Same class          │ Parent-Child classes  │
│ Method name      │ Same               │ Same                 │
│ Parameters       │ Must be different  │ Must be same         │
│ Return type      │ Can differ         │ Same or covariant    │
│ Access modifier  │ Can differ         │ Can only broaden     │
│ @Override        │ Not applicable     │ Recommended ✅        │
│ Inheritance      │ Not required       │ Required             │
│ Polymorphism     │ Static             │ Dynamic              │
│ Resolved by      │ Compiler           │ JVM at runtime       │
└──────────────────┴─────────────────────┴──────────────────────┘
```

## ⚠️ What CANNOT Be Overridden

```java
class Parent {
    // 1. final method
    final void show() { }           // ❌ cannot override

    // 2. static method — HIDDEN not overridden
    static void display() { }       // ❌ hiding, not overriding

    // 3. private method — not visible
    private void secret() { }       // ❌ cannot override

    // 4. constructor — not inherited
}
```

## 🔄 Object Type vs Reference Type

```java
// Reference type = Animal (left side)
// Object type    = Dog    (right side)
Animal a = new Dog();

// Compile-time: compiler only sees Animal reference
// Runtime:      JVM sees actual Dog object

a.sound();      // ✅ sound() in Animal → overridden → Dog.sound()!
a.bark();       // ❌ COMPILE ERROR! bark() not in Animal reference
                // compiler only checks Animal — doesn't see Dog methods!

// To access Dog methods — cast!
Dog d = (Dog) a;
d.bark();       // ✅ now can call bark()

// ⚠️ Wrong cast → ClassCastException at runtime!
Animal a2 = new Cat();
Dog d2 = (Dog) a2; // compiles ✅ but 💥 ClassCastException at runtime!

// Safe cast — always check first!
if (a instanceof Dog dog) {
    dog.bark(); // ✅ safe (Java 16+ pattern matching)
}
```

## 🔄 Polymorphism with Constructors ⚠️

```java
class Parent {
    Parent() {
        show();         // ⚠️ calling overridable method in constructor!
    }
    void show() {
        System.out.println("Parent show");
    }
}

class Child extends Parent {
    int x = 10;

    @Override
    void show() {
        System.out.println("Child show: " + x);
    }
}

new Child();
// Output: "Child show: 0" 😱

// Why 0? Not 10?
// 1. Child() called
// 2. super() runs first → Parent constructor
// 3. show() called → polymorphism → Child.show()!
// 4. BUT x not yet initialized! (Child fields init AFTER super())
// 5. x = 0 (default) ← dangerous!
// Rule: NEVER call overridable methods in constructors! ⚠️
```

## 🎯 Interview Cheat Sheet

```
Q: What is Polymorphism?
A: One name, many forms. Same method behaves
   differently based on object type or parameters.

Q: How many types of polymorphism in Java?
A: Two:
   1. Compile-time (Static)  → Method Overloading
   2. Runtime (Dynamic)      → Method Overriding

Q: Can we overload main() method?
A: YES ✅ — but JVM calls only String[] args version.
   public static void main(int[] args) {} ← valid but JVM ignores

Q: Can we override main() method?
A: NO ❌ — main() is static → can be hidden, not overridden.

Q: What is the role of @Override annotation?
A: Optional but HIGHLY recommended.
   Tells compiler "this is an override."
   If method signature wrong → compile error catches it!
   Without @Override → silent bug (new method, not override).

Q: Can we overload by changing return type only?
A: NO ❌ — compile error. Parameters must differ.

Q: Difference between method hiding and overriding?
A: Overriding → instance methods, runtime decision
   Hiding     → static methods, compile-time decision

Q: Can constructors be overloaded?
A: YES ✅ — very common! Constructor overloading.
   Same class name, different parameters.

Q: Can constructors be overridden?
A: NO ❌ — constructors NOT inherited → cannot override.

Q: Automatic type promotion in overloading?
A: YES — if exact match not found, Java promotes:
   byte→short→int→long→float→double
   char→int

Q: Which polymorphism gives better performance?
A: Compile-time → slightly faster (resolved at compile time).
   Runtime → slight overhead (vtable lookup at runtime).
   In practice — difference negligible.

Q: Can private methods be overloaded?
A: YES ✅ — overloading within same class.
   But private methods CANNOT be overridden.
```

## 🧩 DSA Angle

```java
// 1. Polymorphism in Graph algorithms
abstract class GraphAlgorithm {
    abstract void execute(int[][] graph, int start);

    // Template — same structure, different execution
    void run(int[][] graph, int start) {
        System.out.println("Starting from: " + start);
        execute(graph, start);  // polymorphic call!
        System.out.println("Done!");
    }
}

class BFS extends GraphAlgorithm {
    @Override
    void execute(int[][] graph, int start) {
        Queue<Integer> queue = new LinkedList<>();
        boolean[] visited = new boolean[graph.length];
        queue.add(start);
        visited[start] = true;
        while (!queue.isEmpty()) {
            int node = queue.poll();
            System.out.print(node + " ");
            for (int i = 0; i < graph[node].length; i++)
                if (graph[node][i] == 1 && !visited[i]) {
                    visited[i] = true;
                    queue.add(i);
                }
        }
    }
}

class DFS extends GraphAlgorithm {
    @Override
    void execute(int[][] graph, int start) {
        boolean[] visited = new boolean[graph.length];
        dfs(graph, start, visited);
    }
    private void dfs(int[][] graph, int node, boolean[] visited) {
        visited[node] = true;
        System.out.print(node + " ");
        for (int i = 0; i < graph[node].length; i++)
            if (graph[node][i] == 1 && !visited[i])
                dfs(graph, i, visited);
    }
}

// Swap algorithm without changing client code!
GraphAlgorithm algo = new BFS();
algo.run(graph, 0); // BFS

algo = new DFS();
algo.run(graph, 0); // DFS — same call! ✅

// 2. Comparator overloading — sort by different criteria
Comparator<int[]> byStart  = (a, b) -> a[0] - b[0];
Comparator<int[]> byEnd    = (a, b) -> a[1] - b[1];
Comparator<int[]> byLength = (a, b) -> (a[1]-a[0]) - (b[1]-b[0]);

Arrays.sort(intervals, byStart);  // sort by start
Arrays.sort(intervals, byEnd);    // sort by end
Arrays.sort(intervals, byLength); // sort by length
// Same sort() call — different behavior via polymorphism! ✅

// 3. Method overloading — clean DSA utility
class ArrayUtils {
    static int sum(int[] arr) {
        int s = 0; for (int x : arr) s += x; return s;
    }
    static long sum(long[] arr) {
        long s = 0; for (long x : arr) s += x; return s;
    }
    static double sum(double[] arr) {
        double s = 0; for (double x : arr) s += x; return s;
    }
}
// Same name — different type handled! ✅
```

## 🏢 Real-Life SDE Usage

```java
// 1. Payment strategy — runtime polymorphism
interface PaymentGateway {
    boolean processPayment(double amount);
    boolean refund(double amount);
}

class RazorpayGateway implements PaymentGateway {
    @Override
    public boolean processPayment(double amount) {
        // Razorpay specific API
        return razorpayClient.charge(amount);
    }
    @Override
    public boolean refund(double amount) {
        return razorpayClient.refund(amount);
    }
}

class StripeGateway implements PaymentGateway {
    @Override
    public boolean processPayment(double amount) {
        // Stripe specific API
        return stripeClient.createCharge(amount);
    }
    @Override
    public boolean refund(double amount) {
        return stripeClient.createRefund(amount);
    }
}

class PaymentService {
    private PaymentGateway gateway; // polymorphic reference!

    PaymentService(PaymentGateway gateway) {
        this.gateway = gateway;
    }

    void checkout(double amount) {
        if (!gateway.processPayment(amount))  // polymorphic call!
            throw new PaymentFailedException();
    }
}

// Switch gateway without changing PaymentService!
PaymentService s1 = new PaymentService(new RazorpayGateway());
PaymentService s2 = new PaymentService(new StripeGateway());

// 2. Method overloading — REST controller
@RestController
class UserController {

    // Overloaded — different param types
    @GetMapping("/users/{id}")
    ResponseEntity<User> getUser(@PathVariable Long id) {
        return ResponseEntity.ok(userService.findById(id));
    }

    @GetMapping("/users")
    ResponseEntity<List<User>> getUser(@RequestParam String email) {
        return ResponseEntity.ok(userService.findByEmail(email));
    }

    @GetMapping("/users/all")
    ResponseEntity<List<User>> getUser(@RequestParam int page,
                                       @RequestParam int size) {
        return ResponseEntity.ok(userService.findAll(page, size));
    }
}

// 3. Logging — polymorphism across environments
abstract class Logger {
    abstract void log(String message);

    // Template
    final void info(String msg)  { log("[INFO]  " + msg); }
    final void error(String msg) { log("[ERROR] " + msg); }
    final void warn(String msg)  { log("[WARN]  " + msg); }
}

class ConsoleLogger extends Logger {
    @Override
    void log(String message) {
        System.out.println(message); // dev environment
    }
}

class FileLogger extends Logger {
    @Override
    void log(String message) {
        fileWriter.write(message); // production
    }
}

class CloudLogger extends Logger {
    @Override
    void log(String message) {
        cloudWatchClient.send(message); // AWS CloudWatch
    }
}

// Inject based on environment — no code change!
Logger logger = isProd ? new CloudLogger() : new ConsoleLogger();
logger.info("Application started"); // same call everywhere! ✅
```
