## 📦 Encapsulation in Java — Complete Guide

> **Encapsulation** means **bundling data (fields) + methods (behavior) together** into a single unit (class) and **controlling access** to that data.

---

## 📌 Core Idea

```text
Without Encapsulation:
Data and methods scattered everywhere
Anyone can access and corrupt anything 😩

With Encapsulation:
┌─────────────────────────────────────┐
│           CLASS (Capsule)           │
│                                     │
│  🔒 private data    ← hidden inside │
│  ✅ public methods  ← controlled    │
│                       gateway       │
└─────────────────────────────────────┘
Outside world ONLY goes through methods
Never touches data directly!
```

<Aside type="tip">
**Key Insight**: Encapsulation is not about secrecy — it's about **controlled access**.  
By hiding internal state and exposing only validated interfaces, you:
- ✅ Prevent invalid state
- ✅ Enable validation logic
- ✅ Allow internal changes without breaking clients
- ✅ Improve maintainability and testability
</Aside>

## 🔬 How It Works — Before & After

<Code lang="java" title="❌ Without Encapsulation — dangerous!" code={`class Student {
    public String name;
    public int age;
    public double marks;
}

Student s = new Student();
s.age   = -100;    // 😱 invalid age — nobody stopped it!
s.marks = 999999;  // 😱 impossible marks — nobody stopped it!
s.name  = null;    // 😱 null name — may cause NPE later!`} />

<Code lang="java" title="✅ With Encapsulation — safe and validated!" code={`class Student {
    // Step 1: Make fields PRIVATE — hidden from outside
    private String name;
    private int age;
    private double marks;

    // Step 2: Provide PUBLIC controlled access

    // Getters — safe read access
    public String getName()  { return name; }
    public int    getAge()   { return age; }
    public double getMarks() { return marks; }

    // Setters — write access WITH validation
    public void setName(String name) {
        if (name == null || name.isEmpty()) {
            throw new IllegalArgumentException("Name cannot be empty!");
        }
        this.name = name;
    }

    public void setAge(int age) {
        if (age < 0 || age > 150) {
            throw new IllegalArgumentException("Invalid age: " + age);
        }
        this.age = age;
    }

    public void setMarks(double marks) {
        if (marks < 0 || marks > 100) {
            throw new IllegalArgumentException("Marks must be 0-100!");
        }
        this.marks = marks;
    }
}

// Usage — validation enforced!
Student s = new Student();
s.setAge(-100);  // ❌ Exception thrown — blocked! ✅
s.setAge(20);    // ✅ Valid — goes through validation
System.out.println(s.getAge());  // ✅ Safe read: 20`} />

## 🔬 In-Depth — What Encapsulation Gives You

<CardGrid>
  <Card title="Benefit 1: Validation" icon="shield">
    ```text
    Setter checks data before storing.
    Bad data NEVER enters your object.
    
    Example:
    public void setAge(int age) {
        if (age < 0 || age > 150) {
            throw new IllegalArgumentException();
        }
        this.age = age;  // Only valid values reach the field
    }
    ```
  </Card>
  
  <Card title="Benefit 2: Flexibility" icon="rocket">
    ```text
    Change internal implementation freely.
    External API stays same — nobody's code breaks.
    
    Example:
    // Version 1: Store as String
    private String phoneNumber;
    public String getPhone() { return phoneNumber; }
    
    // Version 2: Refactor to object — no API change!
    private PhoneNumber phoneNumber;
    public String getPhone() { 
        return phoneNumber.format();  // Same public API ✅
    }
    ```
  </Card>

  <Card title="Benefit 3: Read-Only / Write-Only Control" icon="information">
    ```text
    Only getter → read-only field (immutable after construction)
    Only setter → write-only field (e.g., password)
    Both → full controlled access
    
    Example:
    private final String id;  // final + no setter = immutable
    public String getId() { return id; }  // Read-only ✅
    ```
  </Card>

  <Card title="Benefit 4: Maintainability" icon="approve-check">
    ```text
    Centralized change — update one place.
    All callers automatically get new behavior.
    
    Example:
    public void setPrice(double price) {
        // Add tax calculation in ONE place
        this.price = price * 1.18;  // All callers get tax automatically
    }
    ```
  </Card>
</CardGrid>

## 🔄 Types of Access via Encapsulation

<Code lang="java" title="Controlled access patterns in practice" code={`class Employee {
    private String id;
    private String name;
    private double salary;
    private String password;

    // 1️⃣ Read-Only — getter only (no setter!)
    public String getId() { return id; }
    // id set once via constructor, never changed → immutable

    // 2️⃣ Write-Only — setter only (no getter!)
    public void setPassword(String pwd) {
        this.password = BCrypt.hash(pwd);  // Hash before store!
    }
    // No getPassword() — password never exposed! ✅ Security

    // 3️⃣ Full Access — getter + setter with validation
    public double getSalary() { return salary; }
    public void setSalary(double salary) {
        if (salary < 0) {
            throw new IllegalArgumentException("Salary can't be negative!");
        }
        this.salary = salary;
    }

    // 4️⃣ Computed — getter calculates, no direct field
    public double getAnnualSalary() {
        return salary * 12;  // Computed — no annualSalary field!
    }

    // Constructor — initialize immutable fields
    public Employee(String id, String name) {
        this.id = id;      // Set once, never changed
        this.name = name;
    }
}`} />

<Aside type="note">
**Design Tip**: Favor **read-only** (immutable) fields where possible.  
Immutable objects are inherently thread-safe, easier to reason about, and prevent accidental mutation.
</Aside>

## 🏗️ Encapsulation with Constructor — Immutable Patterns

<Code lang="java" title="BankAccount — encapsulated with constructor validation" code={`class BankAccount {
    private final long accountNumber;  // final → immutable after construction
    private double balance;            // mutable via controlled methods
    private final String owner;        // final → immutable

    // Constructor — sets initial state safely with validation
    public BankAccount(long accNo, String owner, double initialDeposit) {
        if (initialDeposit < 0) {
            throw new IllegalArgumentException("Initial deposit can't be negative!");
        }
        if (owner == null || owner.isEmpty()) {
            throw new IllegalArgumentException("Owner name required!");
        }
        this.accountNumber = accNo;  // Set once, never changed
        this.owner = owner;           // Set once, never changed
        this.balance = initialDeposit;
    }

    // Read-only access for immutable fields
    public long getAccountNumber() { return accountNumber; }
    public String getOwner() { return owner; }

    // Read access for mutable field
    public double getBalance() { return balance; }

    // Controlled mutation — deposit with validation
    public void deposit(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("Deposit must be positive!");
        }
        balance += amount;
    }

    // Controlled mutation — withdraw with validation + business rule
    public void withdraw(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("Amount must be positive!");
        }
        if (amount > balance) {
            throw new IllegalStateException("Insufficient balance!");
        }
        balance -= amount;
    }

    // Computed property — no direct field
    public boolean isOverdrawn() {
        return balance < 0;
    }
}`} />

<Aside type="caution">
**Immutable Field Pattern**:  
Use `private final` + **no setter** + **constructor initialization** for fields that should never change after object creation.  
This creates **immutable objects** — thread-safe and easier to reason about.
</Aside>

## ⚠️ Common Encapsulation Pitfalls

<CardGrid>
  <Card title="Pitfall 1: Exposing Mutable Collections" icon="error">
    ```java
    class Team {
        private List<String> members = new ArrayList<>();
        
        // ❌ Dangerous: Caller can modify internal list!
        public List<String> getMembers() {
            return members;  // External code can add/remove!
        }
    }
    
    // Usage:
    Team team = new Team();
    team.getMembers().add("Hacker");  // 😱 Bypasses business logic!
    
    // ✅ Safe alternatives:
    // Option 1: Return unmodifiable view
    public List<String> getMembers() {
        return Collections.unmodifiableList(members);
    }
    
    // Option 2: Return defensive copy
    public List<String> getMembers() {
        return new ArrayList<>(members);
    }
    
    // Option 3: Provide controlled add/remove methods
    public void addMember(String name) { /* validation */ }
    public void removeMember(String name) { /* validation */ }
    ```
  </Card>
  
  <Card title="Pitfall 2: Returning Mutable Objects Directly" icon="error">
    ```java
    class Event {
        private Date startTime;  // java.util.Date is mutable!
        
        // ❌ Dangerous: Caller can change internal state
        public Date getStartTime() {
            return startTime;  // Caller: event.getStartTime().setTime(...)
        }
    }
    
    // ✅ Safe alternatives:
    // Option 1: Return defensive copy
    public Date getStartTime() {
        return new Date(startTime.getTime());
    }
    
    // Option 2: Use immutable types (Java 8+)
    private Instant startTime;  // Instant is immutable ✅
    public Instant getStartTime() { return startTime; }  // Safe to return directly
    ```
  </Card>

  <Card title="Pitfall 3: Over-Encapsulation — Getters/Setters for Everything" icon="caution">
    ```java
    // ❌ Over-engineered: Simple data holder doesn't need full encapsulation
    class Point {
        private int x;
        private int y;
        public int getX() { return x; }
        public void setX(int x) { this.x = x; }
        public int getY() { return y; }
        public void setY(int y) { this.y = y; }
    }
    
    // ✅ Better: Immutable value object with constructor
    public final class Point {
        private final int x;
        private final int y;
        
        public Point(int x, int y) {
            this.x = x;
            this.y = y;
        }
        public int x() { return x; }  // Minimal getter (no "get" prefix for value objects)
        public int y() { return y; }
        // No setters — immutable ✅
    }
    ```
  </Card>
</CardGrid>

## 🎯 Interview Cheat Sheet

<CardGrid>
  <Card title="Q: What is Encapsulation?" icon="information">
    **Bundling data (fields) and methods together in a class** and **restricting direct access** to internal data using `private` + public getters/setters.  
    ✅ Focus: Controlled access, validation, maintainability.
  </Card>
  
  <Card title="Q: How is Encapsulation achieved in Java?" icon="approve-check">
    1. Make fields `private`  
    2. Provide public getters/setters with validation  
    3. Use constructors to initialize immutable state  
    4. Return defensive copies/unmodifiable views for mutable fields
  </Card>

  <Card title="Q: Difference between Encapsulation and Abstraction?" icon="information">
    - **Encapsulation**: Data + methods bundled + access control → **HOW it's stored** is hidden  
    - **Abstraction**: Implementation logic hidden → **HOW it works** is hidden  
    ✅ They complement each other — encapsulation enables abstraction.
  </Card>

  <Card title="Q: Can a class be fully encapsulated?" icon="approve-check">
    **YES ✅** — all fields `private`, no public fields whatsoever.  
    Every access through methods. Maximum protection.  
    Example: `String` class — all fields private, only public methods exposed.
  </Card>

  <Card title="Q: What if a class has only getters — no setters?" icon="information">
    Class becomes **IMMUTABLE** — object state can't change after creation.  
    ✅ Benefits: Thread-safe, safe to share, predictable behavior.  
    Example: `String`, `Integer`, `LocalDate` — all immutable.
  </Card>

  <Card title="Q: Is it mandatory to have both getter and setter?" icon="approve-check">
    **NO ❌** — depends on design:  
    - Read-only → getter only (e.g., `getId()`)  
    - Write-only → setter only (e.g., `setPassword()`)  
    - Full access → both with validation
  </Card>

  <Card title="Q: Can we encapsulate static fields?" icon="approve-check">
    **YES ✅** — same pattern applies:
    ```java
    private static int count = 0;
    public static int getCount() { return count; }
    public static void increment() { count++; }  // Controlled mutation
    ```
  </Card>

  <Card title="Q: What is tight encapsulation?" icon="shield">
    **ALL fields private, NO public fields whatsoever**.  
    Every access through methods. Maximum protection.  
    ✅ Preferred in production code — minimizes surface area for bugs.
  </Card>

## 🧩 DSA Angle — Encapsulation in Data Structures

<CardGrid>
  <Card title="Pattern: Encapsulated LinkedList — clean and safe" icon="rocket">
    ```java
    class LinkedList<T> {
        // Hidden internal node class — not exposed to clients
        private class Node {
            T data;
            Node next;
            Node(T data) { this.data = data; }
        }

        // Hidden internal state — encapsulated
        private Node head;
        private int size;

        // Controlled operations ONLY — public API
        public void addFirst(T data) {
            Node node = new Node(data);
            node.next = head;
            head = node;
            size++;
        }

        public T removeFirst() {
            if (head == null) {
                throw new NoSuchElementException("List is empty!");
            }
            T data = head.data;
            head = head.next;
            size--;
            return data;
        }

        // Read-only accessors — no direct field exposure
        public int size() { return size; }
        public boolean isEmpty() { return size == 0; }

        // ❌ Nobody can touch head, size, Node directly
        // ✅ Only through controlled, validated methods
    }
    ```
  </Card>
  
  <Card title="Pattern: Encapsulated Graph for Algorithm Reuse" icon="shield">
    ```java
    abstract class Graph {
        // Hidden internal representation — could be adjacency list, matrix, etc.
        protected final int vertices;
        
        protected Graph(int vertices) {
            this.vertices = vertices;
        }
        
        // Abstract methods — concrete implementations provide storage
        public abstract void addEdge(int src, int dest);
        public abstract List<Integer> getNeighbors(int vertex);
        
        // Concrete algorithm — works for ANY Graph implementation
        public List<Integer> bfs(int start) {
            boolean[] visited = new boolean[vertices];
            Queue<Integer> queue = new LinkedList<>();
            List<Integer> result = new ArrayList<>();
            
            queue.offer(start);
            visited[start] = true;
            
            while (!queue.isEmpty()) {
                int vertex = queue.poll();
                result.add(vertex);
                
                // Polymorphic call — works with any Graph implementation
                for (int neighbor : getNeighbors(vertex)) {
                    if (!visited[neighbor]) {
                        visited[neighbor] = true;
                        queue.offer(neighbor);
                    }
                }
            }
            return result;
        }
    }
    ```
  </Card>
</CardGrid>

<Aside type="tip">
**DSA Pro Tip**: Encapsulation enables **algorithm reuse**.  
- Write BFS/DFS once against `Graph` interface — works with adjacency list, matrix, or custom representations  
- Hide internal node structure — clients can't corrupt the data structure  
- Validate operations — prevent invalid graph states (e.g., self-loops if not allowed)
</Aside>

    public final class AppConfig {
        // All fields final + private → immutable
        private final String apiUrl;
        private final int timeoutMs;
        private final boolean enableCache;
        private final Map<String, String> headers;

        // Private constructor — force use of Builder
        private AppConfig(Builder builder) {
            this.apiUrl = Objects.requireNonNull(builder.apiUrl);
            this.timeoutMs = builder.timeoutMs;
            this.enableCache = builder.enableCache;
            // Defensive copy of mutable map
            this.headers = Collections.unmodifiableMap(
                new HashMap<>(builder.headers)
            );
        }

        // Getters — all immutable types or unmodifiable views
        public String apiUrl() { return apiUrl; }
        public int timeoutMs() { return timeoutMs; }
        public boolean enableCache() { return enableCache; }
        public Map<String, String> headers() { return headers; }

        // Builder for flexible, validated construction
        public static class Builder {
            private String apiUrl;  // Required
            private int timeoutMs = 30_000;  // Default
            private boolean enableCache = true;  // Default
            private final Map<String, String> headers = new HashMap<>();

            public Builder apiUrl(String url) {
                this.apiUrl = url; return this;
            }
            public Builder timeoutMs(int ms) {
                if (ms <= 0) throw new IllegalArgumentException();
                this.timeoutMs = ms; return this;
            }
            public Builder header(String name, String value) {
                this.headers.put(name, value); return this;
            }
            public AppConfig build() {
                if (apiUrl == null) throw new IllegalStateException("apiUrl required");
                return new AppConfig(this);
            }
        }
    }

    // Domain layer — pure business logic, no infrastructure concerns
    public interface UserRepository {
        Optional<User> findById(UserId id);
        User save(User user);
        void delete(UserId id);
    }

    // Infrastructure layer — implements persistence details (hidden)
    @Repository
    class JpaUserRepository implements UserRepository {
        @PersistenceContext
        private EntityManager em;

        @Override
        public Optional<User> findById(UserId id) {
            // JPA/Hibernate logic hidden here — encapsulated
            return Optional.ofNullable(em.find(User.class, id));
        }

        @Override
        public User save(User user) {
            // Transaction management, caching, etc. — encapsulated
            return em.merge(user);
        }
    }

    // Service layer — depends on abstraction, not concrete repository
    @Service
    class UserService {
        private final UserRepository users;  // Interface!

        @Autowired
        public UserService(UserRepository users) {
            this.users = users;  // Dependency injection
        }

        public User register(String email, String password) {
            // Business logic — independent of persistence tech
            User user = User.create(email, password);  // Domain logic
            return users.save(user);  // Persistence via interface
        }
    }

## 🔑 Quick Reference Summary

### Encapsulation Checklist

| Scenario | Recommended Approach |
|----------|---------------------|
| Simple data holder | `record` (Java 14+) or immutable class |
| Mutable domain object | Private fields + validated setters + business methods |
| Configuration | Immutable + Builder pattern |
| Internal helper | `private` or package-private |
| Extension point | `protected` method + JavaDoc contract |
| Public API | `public` methods with clear contracts + validation |
| Mutable collection field | Return `unmodifiableList()` or defensive copy |
| Mutable object field | Return defensive copy or use immutable type |

### Access Control Patterns

| Pattern | Fields | Methods | Use Case |
|---------|--------|---------|----------|
| **Read-Only** | `private final` | Getter only | Immutable IDs, timestamps |
| **Write-Only** | `private` | Setter only | Passwords, secrets |
| **Full Access** | `private` | Getter + validated setter | Mutable business data |
| **Computed** | None (or private) | Getter only (calculates) | Derived values, aggregates |
| **Controlled Mutation** | `private` | Business methods (no raw setters) | State machines, workflows |

### Common Pitfalls & Fixes

| Pitfall | Risk | Fix |
|---------|------|-----|
| Public mutable field | Invalid state, thread issues | Make `private` + validated setter |
| Getter returns mutable collection | Caller modifies internal state | Return `unmodifiableList()` or copy |
| Setter accepts mutable object without copy | Caller retains reference, can mutate later | Defensive copy in setter |
| Constructor leaks `this` | Methods called on partially-constructed object | Register listeners after construction |
| Over-encapsulation (getters for everything) | Anemic domain model, logic leakage | Expose behavior, not just data |

<Aside type="caution">
**Final Checklist for Encapsulation Design**:
1. ✅ Start with `private` — relax access only when necessary
2. ✅ Validate inputs at public boundaries (constructors, setters, methods)
3. ✅ Return immutable views or defensive copies for mutable state
4. ✅ Prefer immutability for value objects, config, DTOs
5. ✅ Encapsulate invariants within the class — don't leak validation to callers
6. ✅ Document class-level invariants in JavaDoc
7. ✅ Test public API behavior — not private implementation details
8. ✅ Use modern features: `record`, `var`, immutable collections where appropriate
</Aside>

<Aside type="tip">
**Pro Interview Strategy** for encapsulation questions:
1. Explain **why** encapsulation matters (validation, flexibility, thread safety)
2. Distinguish **access levels** and when to use each (`private` → `public`)
3. Discuss **mutable vs immutable** design tradeoffs
4. Mention **modern Java features** (`record`, immutable collections) that simplify encapsulation
5. Give **real-world examples** (domain objects, config, repositories) to show practical understanding
6. Address **testing concerns** — test behavior via public API, not private implementation

This demonstrates both theoretical knowledge and production-ready design thinking! 🎯
