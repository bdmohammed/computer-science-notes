## 1️⃣ Increment & Decrement Operators

```text
++  → increment by 1
--  → decrement by 1

Two flavors:
```

### 🔄 Pre vs Post — Visual Flow

```text
Pre (++i)              Post (i++)
  |                       |
  v                       v
[Add 1] ──────────▶ [Use Value]
  |                       |
  v                       v
[Use Value]         [Add 1]
```

<Code lang="java" title="Pre vs Post behavior" code={`int x = 5;

// Pre-increment: modify FIRST, then use
int a = ++x;  // x becomes 6 FIRST → a = 6
System.out.println(x); // 6
System.out.println(a); // 6

// Post-increment: use FIRST, then modify
int b = x++;  // b gets 6 FIRST → THEN x becomes 7
System.out.println(x); // 7
System.out.println(b); // 6`} />

### 📊 Expression Behavior Table

| Expression | Initial `x` | Value of `y` | Final `x` | Explanation |
|------------|-------------|--------------|-----------|-------------|
| `y = ++x`  | 10 | 11 | 11 | Pre-increment: `x` becomes 11, then assigned to `y` |
| `y = x++`  | 10 | 10 | 11 | Post-increment: `x` (10) assigned to `y`, then `x` becomes 11 |
| `y = --x`  | 10 | 9 | 9 | Pre-decrement: `x` becomes 9, then assigned to `y` |
| `y = x--`  | 10 | 10 | 9 | Post-decrement: `x` (10) assigned to `y`, then `x` becomes 9 |

## ⚠️ Rules — Must Know

<CardGrid>
  <Card title="Rule 1: Only for Variables, Not Constants" icon="error">
    Increment/decrement operators require a **modifiable variable**.
    
    ```java
    // ✅ Valid: variable
    int x = 4;
    int y = ++x;
    System.out.println(y); // 5
    
    // ❌ Invalid: constant literal
    int y = ++4;  // ❌ C.E: unexpected type
                  // required: variable
                  // found: value
    ```
  </Card>
  
  <Card title="Rule 2: No Nesting Allowed" icon="error">
    You cannot nest `++` or `--` operators.
    
    ```java
    class Test {
        public static void main(String[] args) {
            int x = 4;
            int y = ++(++x);  // ❌ C.E: unexpected type
                              // required: variable
                              // found: value (++x is a value, not a variable)
        }
    }
    // Reason: ++x evaluates to a value (int), not a variable reference.
    // You cannot apply ++ to a temporary value.
    ```
  </Card>

  <Card title="Rule 3: Cannot Apply to final Variables" icon="error">
    `final` variables are immutable — incrementing them violates their contract.
    
    ```java
    class Test {
        public static void main(String[] args) {
            final int x = 4;
            x++;  // ❌ C.E: cannot assign a value to final variable 'x'
            System.out.println(x);
        }
    }
    // Reason: x++ expands to x = x + 1, which is an assignment.
    // final variables cannot be reassigned after initialization.
    ```
  </Card>

  <Card title="Rule 4: Applies to All Primitives Except boolean" icon="approve-check">
    ```java
    // ✅ Valid for numeric primitives
    int x = 10; x++; System.out.println(x); // 11
    char ch = 'a'; ch++; System.out.println(ch); // 'b'
    double d = 10.5; d++; System.out.println(d); // 11.5
    
    // ❌ Invalid for boolean
    boolean b = true;
    b++;  // ❌ C.E: operator ++ cannot be applied to boolean
    ```
    **Why?** `boolean` represents logical state, not numeric value — incrementing has no semantic meaning.
  </Card>
</CardGrid>

## 🔬 Critical Distinction: `b++` vs `b = b + 1`

### The Type Promotion Trap

When arithmetic operators are applied between two operands, the result type is:
```
max(int, type_of_a, type_of_b)
```
This means **`byte + byte` produces `int`**, requiring explicit cast to assign back to `byte`.

<Code lang="java" title="❌ Arithmetic assignment fails without cast" code={`byte a = 10;
byte b = 20;
byte c = a + b;  // ❌ C.E: possible loss of precision
                 // found: int
                 // required: byte

// ✅ Fix with explicit cast:
byte c = (byte)(a + b);`} />

<Code lang="java" title="❌ Same issue with b = b + 1" code={`byte b = 10;
b = b + 1;  // ❌ C.E: possible loss of precision
            // found: int
            // required: byte

// ✅ Fix:
b = (byte)(b + 1);`} />

### ✅ Why `b++` Works Automatically

<Code lang="java" title="Increment operators perform implicit cast" code={`byte b = 10;
b++;  // ✅ Compiles! No explicit cast needed
System.out.println(b); // 11

// Internally, compiler transforms:
// b++  →  b = (byte)(b + 1);
// The cast to original type is inserted AUTOMATICALLY`} />

<Aside type="tip">
**Memory Rule**:  
`b++` is **not** exactly equivalent to `b = b + 1`.  
The increment operator includes an **implicit narrowing cast** back to the variable's original type. This is a special compiler behavior for `++`/`--` only.
</Aside>

## 🎯 Classic Interview Traps

<CardGrid>
  <Card title="Trap 1: The i = i++ Paradox" icon="error">
    ```java
    int i = 10;
    i = i++;  // What is i after this?
    
    // Step-by-step evaluation:
    // 1. i++ is post-increment: 
    //    - temp = i (temp = 10)
    //    - i = i + 1 (i becomes 11)
    //    - expression value = temp (10)
    // 2. Assignment: i = 10 (overwrites the 11!)
    
    System.out.println(i); // 10 😱 NOT 11!
    
    // ✅ To actually increment: use ++i or just i++;
    i = ++i;  // i = 11 ✅
    // OR simply:
    i++;      // i = 11 ✅
    ```
    **Interview Answer**: Post-increment returns the *original* value, which then overwrites the incremented value during assignment.
  </Card>
  
  <Card title="Trap 2: Mixed Pre/Post in Expression" icon="error">
    ```java
    int x = 5;
    System.out.println(x++ + ++x);
    
    // Evaluation order (left-to-right for operands):
    // 1. x++ (post): 
    //    - use current x (5) as operand value
    //    - then x becomes 6
    // 2. ++x (pre):
    //    - x becomes 7 first
    //    - then use x (7) as operand value
    // 3. Add: 5 + 7 = 12
    // Final x = 7
    
    System.out.println(x); // 7
    // Output: 12
    ```
    **Key Insight**: Operand evaluation is **left-to-right**, but pre/post determines *when* the increment happens relative to *using* the value.
  </Card>

  <Card title="Trap 3: Chained Increments in Array Index" icon="error">
    ```java
    int[] arr = {10, 20, 30};
    int i = 0;
    int val = arr[i++] + arr[i];
    
    // Step 1: i++ → use i=0 for arr[0]=10, then i becomes 1
    // Step 2: arr[i] → arr[1] = 20
    // Result: 10 + 20 = 30
    // Final i = 1
    
    System.out.println(val); // 30
    System.out.println(i);   // 1
    ```
    **Pro Tip**: Avoid complex expressions with side effects. Break them into separate statements for clarity:
    ```java
    int first = arr[i++];
    int second = arr[i];
    int val = first + second;
    ```
  </Card>
</CardGrid>

## 🧩 DSA Angle: When to Use Which

<CardGrid>
  <Card title="Pattern: Loop Counters" icon="rocket">
    Post-increment is conventional in `for` loops (though functionally identical to pre-increment for primitives):
    ```java
    // Standard pattern
    for (int i = 0; i < n; i++) {  // i++ is idiomatic
        process(arr[i]);
    }
    
    // Pre-increment also works:
    for (int i = 0; i < n; ++i) {  // Same behavior for int
        process(arr[i]);
    }
    ```
    **Note**: For iterator objects (`List.iterator()`), `++it` (pre) can be marginally more efficient than `it++` (post) because post-increment may create a temporary copy. But for primitives, **no performance difference**.
  </Card>
  
  <Card title="Pattern: Two-Pointer Techniques" icon="shield">
    Use post-increment when you need the *current* index before advancing:
    ```java
    // Merge two sorted arrays
    int i = 0, j = 0, k = 0;
    while (i < left.length && j < right.length) {
        // Use current values, then increment
        result[k++] = (left[i] <= right[j]) ? left[i++] : right[j++];
    }
    // k++, i++, j++ all use post-increment: assign first, then advance pointer
    ```
  </Card>

  <Card title="Pattern: Avoid Side Effects in Complex Expressions" icon="backspace">
    **Interview Best Practice**: Never write code like `arr[i++] + arr[i++]`. It's hard to read and error-prone.
    ```java
    // ❌ Hard to reason about:
    int sum = arr[i++] + arr[i++];
    
    // ✅ Clear and maintainable:
    int first = arr[i++];
    int second = arr[i++];
    int sum = first + second;
    ```
    **Rule of Thumb**: If an expression has more than one side effect (`++`, `--`, assignment), split it into multiple statements.
  </Card>
</CardGrid>
