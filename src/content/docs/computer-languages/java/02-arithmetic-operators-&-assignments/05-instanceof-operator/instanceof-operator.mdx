## ЁЯФН instanceof Operator

> Checks if an object is an **instance of a class, subclass, or interface** at runtime.

```java
// Syntax: reference instanceof Type
String s = "Hello";
System.out.println(s instanceof String);  // true
System.out.println(s instanceof Object);  // true (String extends Object)

Object obj = new Integer(5);
System.out.println(obj instanceof Integer);  // true
System.out.println(obj instanceof String);   // false
```

### ЁЯза How It Works

```mermaid
flowchart TB
    A["r instanceof X"]:::box
    A --> B[object reference]
    A --> C[class / interface / array type]
    
    classDef box stroke:#000,stroke-width:2px,fill:#fff,color:#000;
```

| Component | Description |
|-----------|-------------|
| **`r`** | Object reference variable (can be `null`) |
| **`X`** | Class name, interface name, or array type |
| **Result** | `boolean`: `true` if `r` refers to an instance of `X` (or subtype), `false` otherwise |

<Aside type="note">
**Key Insight**: `instanceof` performs a **runtime type check** based on the **actual object type**, not the reference type. This enables safe downcasting in polymorphic code.
</Aside>

## ЁЯФм Inheritance & Interface Rules

### Class Hierarchy Example

```java
class Animal {}
class Dog extends Animal {}
class Cat extends Animal {}

Animal a1 = new Dog();
Animal a2 = new Cat();

System.out.println(a1 instanceof Dog);    // true  тЬЕ (actual object is Dog)
System.out.println(a1 instanceof Animal); // true  тЬЕ (Dog IS-A Animal)
System.out.println(a1 instanceof Cat);    // false тЭМ (Dog is NOT-A Cat)
System.out.println(a2 instanceof Dog);    // false тЭМ (Cat is NOT-A Dog)
```

### Interface Implementation Example

```java
interface Flyable { void fly(); }
class Bird implements Flyable { public void fly() { } }
class Airplane implements Flyable { public void fly() { } }

Flyable f = new Bird();

System.out.println(f instanceof Flyable);  // true тЬЕ
System.out.println(f instanceof Bird);     // true тЬЕ
System.out.println(f instanceof Airplane); // false тЭМ
```

### ЁЯзм Thread Example (From Notes)

```java
// Java's Thread class: public class Thread extends Object implements Runnable

Thread t = new Thread();

System.out.println(t instanceof Thread);    // true тЬЕ
System.out.println(t instanceof Object);    // true тЬЕ (all classes extend Object)
System.out.println(t instanceof Runnable);  // true тЬЕ (Thread implements Runnable)
```

```mermaid
classDiagram
    class Object
    class Thread    class Runnable

    Thread --|> Object : extends
    Thread ..|> Runnable : implements
```

<Aside type="tip">
**Memory Aid**: `instanceof` follows the **IS-A relationship**.  
If `Dog IS-A Animal`, then `dogRef instanceof Animal` тЖТ `true`.  
If no inheritance/implementation relationship exists, the compiler rejects the check.
</Aside>

## тЪая╕П Compile-Time Type Compatibility

> To use `instanceof`, there **must be a possible type relationship** between the reference type and the checked type. Otherwise тЖТ **Compile Error: "inconvertible types"**.

<Code lang="java" title="тЬЕ Valid: Related types" code={`Thread t = new Thread();

System.out.println(t instanceof String);   // тЭМ C.E: inconvertible types
// Found: java.lang.Thread
// Required: java.lang.String
// Reason: Thread and String have no inheritance relationship тЖТ compiler rejects`} />

<Code lang="java" title="тЬЕ Valid: Compatible hierarchy" code={`Object o = new String("hello");

System.out.println(o instanceof String);   // true тЬЕ (String IS-A Object)
System.out.println(o instanceof Object);   // true тЬЕ
System.out.println(o instanceof Thread);   // false тЬЕ (compiles, but runtime false)`} />

<Aside type="caution">
**Compiler vs. Runtime**:  
- **Compile time**: Checks if types *could* be related (inheritance/interface). If not тЖТ C.E.  
- **Runtime**: Checks if the *actual object* is an instance of the type. Returns `true`/`false`.
</Aside>

## ЁЯОп null instanceof тАФ Never Throws NPE

> **`null instanceof X` always returns `false`** тАФ no `NullPointerException`, ever.

<Code lang="java" title="null is not an instance of anything" code={`String s = null;
System.out.println(s instanceof String);  // false тЬЕ (NOT NPE!)

Object obj = null;
System.out.println(obj instanceof Object);  // false тЬЕ
System.out.println(obj instanceof Runnable); // false тЬЕ
// Universal rule:
// For ANY type X (class, interface, array): null instanceof X тЖТ false`} />

<Aside type="tip">
**Safe Null Check Pattern**: Use `instanceof` as a combined null + type check:
```java
if (obj instanceof String s) {  // Java 16+ pattern matching
    // s is non-null String here тАФ no extra null check needed!
    System.out.println(s.length());
}
// If obj is null or not a String, block is skipped safely.
```
</Aside>

## ЁЯФД Parent Reference vs. Child Type Check

> Checking if a **parent-type reference** actually holds a **child-type object**.

<Code lang="java" title="Parent reference, child object тЖТ true" code={`Object o = new String("ashok");  // Reference type: Object, Actual object: String

System.out.println(o instanceof String);  // true тЬЕ
// Runtime checks actual object type, not reference type

if (o instanceof String) {
    String s = (String) o;  // Safe downcast
    System.out.println(s.toUpperCase());  // "ASHOK"
}`} />

<Code lang="java" title="Parent object, child type check тЖТ false" code={`Object o = new Object();  // Actual object is plain Object

System.out.println(o instanceof String);  // false тЬЕ
// Object is NOT a String, so check fails

// This is why instanceof is essential before downcasting:
if (o instanceof String) {
    String s = (String) o;  // This block won't execute тЖТ safe!
}`} />

<Aside type="danger">
**Downcasting Without instanceof тЖТ ClassCastException**:
```java
Object o = new Object();
String s = (String) o;  // ЁЯТе Runtime: ClassCastException!

// тЬЕ Always guard with instanceof (or pattern matching):
if (o instanceof String s) {
    // Safe to use s here
}```
</Aside>

## ЁЯЯб Pattern Matching for instanceof (Java 16+)

> Eliminates manual casting by introducing a **pattern variable** automatically bound to the cast type.

### Before Java 16: Manual Cast

<Code lang="java" title="Verbose pre-Java-16 style" code={`Object obj = getSomething();

if (obj instanceof String) {
    String s = (String) obj;  // Manual cast тАФ repetitive!
    System.out.println(s.length());
    System.out.println(s.toUpperCase());
} else if (obj instanceof Integer) {
    Integer i = (Integer) obj;  // Another manual cast
    System.out.println(i * 2);
}`} />

### Java 16+: Pattern Matching

<Code lang="java" title="Clean pattern matching syntax" code={`Object obj = getSomething();

// Pattern variable 's' is automatically cast and scoped to the if-block
if (obj instanceof String s) {
    System.out.println(s.length());      // s is String тАФ no cast needed!
    System.out.println(s.toUpperCase());
} 
// Enhanced: if-else chain with pattern matching
else if (obj instanceof Integer i) {
    System.out.println(i * 2);  // i is Integer
}
// s and i are not accessible outside their respective blocks тЬЕ`} />

### ЁЯФС Pattern Matching Rules

| Feature | Detail |
|---------|--------|
| **Syntax** | `ref instanceof Type varName` |
| **Scope** | `varName` is only in scope inside the `if`/`else` block where condition is `true` |
| **Null Safety** | If `ref` is `null`, condition is `false` тАФ block skipped, no NPE |
| **Flow Typing** | In `if-else` chains, later branches know earlier patterns failed |

<Code lang="java" title="Flow typing example" code={`void process(Object obj) {
    if (obj instanceof String s && s.length() > 5) {
        // s is String AND length > 5
        System.out.println("Long string: " + s);    } else if (obj instanceof String s) {
        // Here, obj IS a String (otherwise first branch would've matched)
        // AND s.length() <= 5 (otherwise first branch would've matched)
        System.out.println("Short string: " + s);
    }
    // s is NOT accessible here тАФ out of scope тЬЕ
}`} />

<Aside type="tip">
**Pro Tip**: Pattern matching works with logical operators:
```java
if (obj instanceof String s && s.startsWith("Hello")) {
    // s is String AND starts with "Hello"
}
// Short-circuit evaluation ensures s is only used when type check passes.
```
</Aside>

<Code lang="java" title="Behavior comparison" code={`class Animal {}
class Dog extends Animal {}

System.out.println(a instanceof Dog);    // true
System.out.println(a instanceof Animal); // true

// getClass(): checks exact class тЖТ false for parent
System.out.println(a.getClass() == Dog.class);    // true
System.out.println(a.getClass() == Animal.class); // false

// isInstance(): reflective instanceof тЖТ same as instanceof
System.out.println(Dog.class.isInstance(a));    // true
System.out.println(Animal.class.isInstance(a)); // true

// Null handling:

System.out.println(nullRef instanceof Animal);        // false тЬЕ
System.out.println(Animal.class.isInstance(nullRef)); // false тЬЕ
// System.out.println(nullRef.getClass() == Animal.class); // ЁЯТе NPE!`} />

## ЁЯОп Interview Cheat Sheet

<CardGrid>
  <Card title="Q: What does null instanceof String return?" icon="approve-check">
    **`false`** тАФ never throws `NullPointerException`.  
    `null` is not an instance of any type.
  </Card>
  
  <Card title="Q: Can instanceof be used with unrelated types?" icon="error">
    **NO тЭМ** тАФ compiler requires a possible inheritance/interface relationship.  
    `threadRef instanceof String` тЖТ Compile Error: "inconvertible types".
  </Card>

  <Card title="Q: What is the result of `new Object() instanceof String`?" icon="information">
    **`false`** тАФ the actual object is `Object`, not `String`.  
    `instanceof` checks runtime type, not reference type.
  </Card>
  <Card title="Q: How does pattern matching improve instanceof?" icon="rocket">
    **Eliminates manual casting** and reduces boilerplate:
```java
    // Before:
    if (obj instanceof String) {
        String s = (String) obj;  // cast
        use(s);
    }
    // After (Java 16+):
    if (obj instanceof String s) {
        use(s);  // auto-cast, s in scope
    }
```
  </Card>

  <Card title="Q: Can instanceof check array types?" icon="approve-check">
    **YES тЬЕ** тАФ arrays are objects with special runtime type info:
```java
    int[] nums = {1, 2, 3};
    System.out.println(nums instanceof int[]);     // true
    System.out.println(nums instanceof Object);    // true (arrays extend Object)
    System.out.println(nums instanceof String[]);  // false
```
  </Card>

  <Card title="Q: Does instanceof work with generics?" icon="caution">
    **NO тЭМ** тАФ due to type erasure:
```java
    List<String> list = new ArrayList<>();
    // System.out.println(list instanceof List<String>);  // тЭМ C.E: generic type not reified
    
    // тЬЕ Check raw type only:
    System.out.println(list instanceof List);  // true
```
  </Card>
</CardGrid>

        }
    }
```
  </Card>
  
  <Card title="Pattern: Visitor-Like Dispatch Without Interfaces" icon="rocket">
    Use `instanceof` for simple type-based behavior when interfaces aren't feasible:
```java
    void render(Component c) {
        if (c instanceof Button b) {
            renderButton(b);
        } else if (c instanceof Label l) {
            renderLabel(l);
        } else if (c instanceof Image img) {
            renderImage(img);
        }
    }
    // Note: Prefer polymorphism (c.render()) when possible тАФ instanceof is a fallback.
```
  </Card>

  <Card title="Pattern: Defensive API Parameter Validation" icon="backspace">
    Validate input types in flexible APIs:
```java
    public void configure(Object config) {
        if (!(config instanceof Map)) {
            throw new IllegalArgumentException("Config must be a Map");
        }
        // Safe to cast after check (or use pattern matching):
        if (config instanceof Map<?, ?> map) {
            // Process map...
        }
    }
```
  </Card>
</CardGrid>

<Aside type="tip">
**Best Practice**: Prefer **polymorphism** over `instanceof` when possible.  
If you find yourself writing long `if (x instanceof A) ... else if (x instanceof B) ...` chains, consider:
- Using interfaces with default methods
- The Visitor pattern
- Strategy pattern with functional interfaces  
`instanceof` is powerful, but overuse can signal missed abstraction opportunities.
</Aside>

## ЁЯФС Quick Reference Summary

| Scenario | `instanceof` Result | Notes |
|----------|---------------------|-------|
| `obj instanceof ActualClass` | `true` | Exact type match |
| `obj instanceof ParentClass` | `true` | Inheritance: child IS-A parent |
| `obj instanceof Interface` | `true` | If class implements interface |
| `null instanceof AnyType` | `false` | Never throws NPE |
| `parentRef instanceof ChildClass` | `false` | Actual object is parent, not child |
| `obj instanceof UnrelatedType` | тЭМ Compile Error | No inheritance relationship |
| `array instanceof ArrayType` | `true`/`false` | Arrays have runtime type info |
| `generic instanceof List<T>` | тЭМ Compile Error | Type erasure prevents generic check |

<Aside type="caution">
**Final Checklist**:
1. тЬЕ `instanceof` checks **runtime type**, not reference type
2. тЬЕ Always returns `boolean` тАФ never throws exception (even for `null`)
3. тЬЕ Compiler enforces **type compatibility** тАФ unrelated types тЖТ C.E
4. тЬЕ Pattern matching (Java 16+) auto-casts and scopes the pattern variable
5. тЬЕ Use `getClass()` for **exact class match** (no inheritance), but handle null
6. тЬЕ Prefer polymorphism over long `instanceof` chains when designing APIs
7. тЬЕ Generics are erased тАФ cannot check `instanceof List<String>`
</Aside>
