## 🔟 Assignment Operators

```text
=     simple assignment
+=    add and assign
-=    subtract and assign
*=    multiply and assign
/=    divide and assign
%=    modulus and assign
&=    bitwise AND assign
|=    bitwise OR assign
^=    bitwise XOR assign
<<=   left shift assign
>>=   right shift assign
>>>=  unsigned right shift assign
```

### 2. Chained Assignment

<Code lang="java" title="✅ Valid: Chained assignment after declaration" code={`int a, b, c, d;
a = b = c = d = 20;  // ✅ All get 20
System.out.println(a + "---" + b + "---" + c + "---" + d);
// Output: 20---20---20---20`} />

<Code lang="java" title="❌ Invalid: Chained assignment at declaration" code={`// Cannot reference variables before they're declared:
int a = b = c = d = 30;  // ❌ C.E: cannot find symbol: variable b
// Reason: b, c, d don't exist yet when evaluating right-to-left`} />
<Aside type="note">
**Evaluation Order**: Chained assignment evaluates **right-to-left**, but all variables must be **declared first**.
</Aside>

### 3. Compound Assignment — The Hidden Cast Trap ⚠️

<Code lang="java" title="Compound operators auto-cast internally" code={`int x = 10;
x += 5;   // Equivalent to: x = (int)(x + 5) → 15
x -= 3;   // x = (int)(x - 3) → 12
x *= 2;   // x = (int)(x * 2) → 24
x /= 4;   // x = (int)(x / 4) → 6
x %= 4;   // x = (int)(x % 4) → 2`} />

#### 🔬 Why `b += 1` Works but `b = b + 1` Fails

<Code lang="java" title="❌ Simple assignment fails due to type promotion" code={`byte b = 10;
b = b + 1;  // ❌ C.E: possible loss of precision
            // b + 1 promotes to int, can't assign int → byte without cast`} />

<Code lang="java" title="✅ Compound assignment includes implicit cast" code={`byte b = 10;
b += 1;  // ✅ Compiles! Equivalent to: b = (byte)(b + 1)
System.out.println(b);  // 11

// This is the SAME behavior as increment operators:
b++;  // ✅ Also auto-casts: b = (byte)(b + 1)`} />

<Aside type="danger">
**Golden Rule**: Compound assignment operators (`+=`, `-=`, etc.) **automatically insert a narrowing cast** to the left-hand type.  
This is why `byteVar += intVal` compiles, but `byteVar = byteVar + intVal` does not.
</Aside>

#### 🧮 Complex Expression Evaluation

<Code lang="java" title="Compound ops with chained assignment" code={`int a, b, c, d;
a = b = c = d = 20;

// Evaluate right-to-left for assignment, but left-to-right for operands:
a += b -= c *= d /= 2;

// Step-by-step:
// 1. d /= 2 → d = 20 / 2 = 10
// 2. c *= d → c = 20 * 10 = 200
// 3. b -= c → b = 20 - 200 = -180
// 4. a += b → a = 20 + (-180) = -160

System.out.println(a + "--" + b + "---" + c + "---" + d);
// Output: -160---180---200---10  (Note: b shows as 180 due to sign in string concat)`} />
<Aside type="caution">
**Avoid Complex Compound Chains**: Expressions like `a += b -= c *= d` are hard to read and debug. Prefer separate statements for clarity.
</Aside>

## 1️⃣1️⃣ Conditional (Ternary) Operator

```text
condition ? valueIfTrue : valueIfFalse
```

<Code lang="java" title="Basic ternary usage" code={`int a = 10, b = 20;
int max = (a > b) ? a : b;  // 20 ✅

// Nested ternary (works but hard to read — prefer if-else):
int x = 5;
String result = x > 0 ? "positive" : x < 0 ? "negative" : "zero";
System.out.println(result);  // "positive"`} />

### ⚠️ Type Promotion in Ternary — Interview Trap

<Code lang="java" title="Ternary result type follows promotion rules" code={`int a = 10, b = 20;

// Both branches are int literals → result is int
byte c1 = (a > b) ? 30 : 40;  // ❌ C.E: possible loss of precision
                              // Ternary result is int, can't assign to byte

// ✅ Fix: Explicit cast
byte c2 = (byte) ((a < b) ? 30 : 40);  // ✅ Works

// ✅ Or: Use byte literals (still promotes to int, but compiler may optimize)
byte c3 = (a < b) ? (byte)30 : (byte)40;  // ✅`} />

<Aside type="note">
**Ternary Type Rule**: The result type of `? :` is determined by **binary numeric promotion** of the two operand types — same rules as arithmetic operators. If either operand is `int` or larger, result is at least `int`.
</Aside>

## 1️⃣2️⃣ `new` Operator — Object Creation

> Creates a **new object on the Heap**, calls constructor, returns reference.

<Code lang="java" title="new operator examples" code={`String s     = new String("Hello");    // String object
int[] arr    = new int[5];               // array object
Student st   = new Student("Alice");     // custom class object

// What 'new' does internally:
// 1. Allocates memory on HEAP for the object// 2. Calls the appropriate constructor
// 3. Returns reference to the newly created object`} />

<Aside type="note">
**No `delete` in Java**: Memory deallocation is handled automatically by the **Garbage Collector**. Developers don't manually free memory.
</Aside>

## 1️⃣3️⃣ Array Access Operator `[]`

> Used for **declaring**, **creating**, and **accessing** array elements.

<Code lang="java" title="Array declaration and access" code={`// Declaration + initialization
int[] arr = {10, 20, 30};

// Access by index (0-based)
System.out.println(arr[0]);  // 10
arr[1] = 99;                 // Modify element

// 2D array (array of arrays)
int[][] matrix = new int[3][3];
matrix[1][2] = 5;  // Row 1, Column 2

// Jagged array (rows of different lengths)
int[][] jagged = new int[2][];
jagged[0] = new int[3];  // Row 0: 3 elements
jagged[1] = new int[5];  // Row 1: 5 elements`} />

<Aside type="danger">
**Bounds Checking**:
```java
arr[-1]           // 💥 ArrayIndexOutOfBoundsException (negative index)
arr[arr.length]   // 💥 ArrayIndexOutOfBoundsException (valid: 0 to length-1)
arr[null]         // 💥 NullPointerException (if arr reference is null)
```
Always validate index: `if (index >= 0 && index < arr.length)`
</Aside>

int result = 2 + 3 * 4;  // 2 + 12 = 14 (NOT 20!)

int result2 = (2 + 3) * 4;  // 5 * 4 = 20 ✅

// Mixed operators: evaluate by precedence, then associativity

int y = x++ + ++x * 2;  
// Step 1: x++ → use 10, x becomes 11
// Step 2: ++x → x becomes 12, use 12
// Step 3: 12 * 2 = 24 (* before +)
// Step 4: 10 + 24 = 34
System.out.println(y);  // 34`} />

## 1️⃣5️⃣ Operand Evaluation Order — Left to Right Always!

> **Critical Distinction**: Operator **precedence** determines *which operation happens first*, but operand **evaluation order** is **always left-to-right**.

<Code lang="java" title="Left-to-right operand evaluation" code={`int a = 1;
int result = ++a + a++ + ++a;

// Evaluation (left-to-right for operands):
// 1. ++a (first operand): a becomes 2, value = 2
// 2. a++ (second operand): value = 2, then a becomes 3
// 3. ++a (third operand): a becomes 4, value = 4
// 4. Now compute: 2 + 2 + 4 = 8

System.out.println(result);  // 8
System.out.println(a);       // 4`} />

### 🧪 Method Call Evaluation Order
<Code lang="java" title="Methods are called left-to-right, regardless of precedence" code={`public class EvalOrder {
    static int m1(int x) {
        System.out.println(x);
        return x;
    }
    
    public static void main(String[] args) {
        // Expression: m1(1) + m1(2) * m1(3) / m1(4) * m1(5) + m1(6)
        // Precedence: * and / before +, but methods called L→R!
        
        int result = m1(1) + m1(2) * m1(3) / m1(4) * m1(5) + m1(6);
        
        // Output order (method calls):
        // 1, 2, 3, 4, 5, 6  ← Always left-to-right!
        
        // Then compute by precedence:
        // 1 + 2*3/4*5 + 6
        // = 1 + 6/4*5 + 6
        // = 1 + 1*5 + 6   (integer division: 6/4 = 1)
        // = 1 + 5 + 6 = 12
        
        System.out.println("Result: " + result);  // 12
    }
}`} />

<Aside type="tip">
**Interview Gold**: When asked about evaluation order:
1. State: *"Operands are always evaluated left-to-right"*
2. Clarify: *"Precedence determines operation order, not evaluation order"*
3. Example: `a() + b() * c()` calls `a()`, then `b()`, then `c()` — even though `*` happens before `+`
</Aside>

## 1️⃣6️⃣ `new` vs `newInstance()` — Object Creation Comparison

<CardGrid>
  <Card title="`new` Operator" icon="rocket">
```java
    // Compile-time object creation
    Student s = new Student();  // ✅ Simple, fast, type-safe
    
    // Characteristics:
    // • Class name known at compile time
    // • Can call ANY constructor (with args)
    // • Checked at compile time
    // • No exceptions thrown (unless constructor throws)
    // • Fast — direct bytecode instruction
```
  </Card>  
  <Card title="`Class.newInstance()` Method" icon="shield">
```java
    // Runtime object creation via Reflection
    Class<?> cls = Class.forName("Student");
    Student s = (Student) cls.newInstance();  // ⚠️ Deprecated Java 9+
    
    // Characteristics:
    // • Class name determined at runtime (e.g., from config)
    // • Can ONLY call NO-ARG constructor
    // • Throws checked exceptions:
    //   - ClassNotFoundException (if .class not found)
    //   - InstantiationException (if no no-arg constructor)
    //   - IllegalAccessException (if constructor not accessible)
    // • Slower — reflection overhead
    // • Type cast required (not type-safe at compile time)
    
    // ✅ Java 9+ preferred way:
    Student s = (Student) cls
                 .getDeclaredConstructor()
                 .newInstance();  // More flexible, better exceptions
```
  </Card>
</CardGrid>

## 1️⃣7️⃣ `instanceof` vs `isInstance()` — Type Checking Comparison

<CardGrid>
  <Card title="`instanceof` Operator" icon="information">
```java
    // Compile-time type check (keyword)
    String s = "Hello";
    System.out.println(s instanceof String);  // true ✅
    
    // Characteristics:
    // • Class/interface known at compile time
    // • Syntax: ref instanceof Type
    // • Returns boolean
    // • null instanceof X → false (never throws NPE)
    // • Compiler checks type compatibility (inconvertible types → C.E)
    // • Java 16+: Pattern matching: if (obj instanceof String s) { ... }
```
  </Card>
  
  <Card title="`Class.isInstance()` Method" icon="backspace">
```java
    // Runtime type check (reflection method)
    String s = "Hello";
    Class<?> cls = Class.forName("java.lang.String");
    System.out.println(cls.isInstance(s));  // true ✅
    
    // Characteristics:
    // • Class determined at runtime (dynamic)
    // • Syntax: classObj.isInstance(ref)
    // • Returns boolean
    // • null → false (never throws NPE)
    // • No compile-time type checking (fully dynamic)
    // • Useful for frameworks, plugins, generic utilities
```
  </Card>
</CardGrid>

### 💻 Dynamic Type Checking Example

<Code lang="java" title="Using isInstance() for runtime type checks" code={`public class DynamicChecker {
    public static void main(String[] args) {
        // args[0] provided at runtime: "Test", "String", "Object", etc.
        try {            Class<?> targetClass = Class.forName(args[0]);
            Object obj = "Hello";  // Test object
            
            // Dynamic type check
            if (targetClass.isInstance(obj)) {
                System.out.println(obj.getClass().getName() + " IS-A " + args[0]);
            } else {
                System.out.println(obj.getClass().getName() + " is NOT-A " + args[0]);
            }
        } catch (ClassNotFoundException e) {
            System.out.println("Class not found: " + args[0]);
        }
    }
}

// Run examples:
// java DynamicChecker String   → "java.lang.String IS-A String" ✅
// java DynamicChecker Integer  → "java.lang.String is NOT-A Integer" ✅
// java DynamicChecker Object   → "java.lang.String IS-A Object" ✅`} />

## 1️⃣8️⃣ `ClassNotFoundException` vs `NoClassDefFoundError`

<CardGrid>
  <Card title="`ClassNotFoundException`" icon="error">
```java
    // ✅ Checked Exception — YOU can catch it
    // Thrown when: Class.forName("name") or ClassLoader.loadClass()
    //              can't find the .class file at runtime
    
    try {
        Class.forName("com.nonexistent.Class");  // Wrong name or missing JAR
    } catch (ClassNotFoundException e) {
        System.out.println("Class not found! Check classpath.");  // ✅ Recoverable
    }
    
    // Common causes:
    // • Typo in class name string
    // • Missing JAR in runtime classpath
    // • Dynamic plugin loading failure
```
  </Card>
  
  <Card title="`NoClassDefFoundError`" icon="error">
```java
    // ❌ Error (unchecked) — JVM throws it, hard to recover
    // Thrown when: Class WAS available at compile time
    //              but MISSING at runtime
        // Example scenario:
    // 1. Compile with: mysql-connector-java.jar in classpath ✅
    // 2. Run WITHOUT: mysql-connector-java.jar in classpath ❌
    // 3. When code tries to use com.mysql.jdbc.Driver → 💥 NoClassDefFoundError
    
    Test t = new Test();  // Compiled fine, but Test.class deleted before run
    // 💥 NoClassDefFoundError: Test
    
    // Common causes:
    // • Deleted/moved .class file after compiling
    // • JAR present at compile, missing at runtime
    // • Circular static initialization dependency
```
  </Card>
</CardGrid>

## 🎯 Master Interview Cheat Sheet

<CardGrid>
  <Card title="Q: What is the result of `i = i++`?" icon="error">
    **Same value** — post-increment stores original value, increments, then assignment overwrites with original.
    ```java
    int i = 10;
    i = i++;  // temp=10, i→11, then i=temp→10
    System.out.println(i);  // 10, NOT 11!
    ```
  </Card>
  
  <Card title="Q: Is `0.1 + 0.2 == 0.3` true?" icon="error">
    **`false`** — floating-point precision issue (IEEE 754 binary representation).  
    ✅ Use `BigDecimal` for exact decimal arithmetic:
    ```java
    BigDecimal a = new BigDecimal("0.1");
    BigDecimal b = new BigDecimal("0.2");
    System.out.println(a.add(b).equals(new BigDecimal("0.3")));  // true
    ```
  </Card>

  <Card title="Q: Difference between `&&` and `&`?" icon="information">
    - `&&`: **Short-circuit** AND — skips right operand if left is `false`  
    - `&`: **Always evaluates both** — bitwise for integers, non-short-circuit logical for booleans  
    ✅ Prefer `&&`/`||` for conditions (safety + performance)
  </Card>

  <Card title="Q: `==` vs `equals()`?" icon="information">
    - `==`: Compares **references** (memory addresses) for objects; **values** for primitives  
    - `equals()`: Compares **content** (defined by class; `String` overrides to compare characters)  
    ✅ Always use `.equals()` for `String` content comparison
  </Card>

  <Card title="Q: What is `null instanceof String`?" icon="approve-check">
    **`false`** — never throws `NullPointerException`.  
    `null` is not an instance of any type. Safe to use in null checks:
    ```java
    if (!(obj instanceof String)) { return; }  // Handles null safely
    ```
  </Card>

  <Card title="Q: `byte b = 10; b += 5` vs `b = b + 5`?" icon="approve-check">
    - `b += 5`: ✅ Compiles — compound op includes **implicit cast**: `b = (byte)(b + 5)`  
    - `b = b + 5`: ❌ Compile error — `b + 5` promotes to `int`, can't assign `int → byte` without explicit cast  
    ✅ Same behavior as `b++` vs `b = b + 1`
  </Card>

  <Card title="Q: `new` vs `newInstance()`?" icon="information">
    - `new`: Compile-time, any constructor, fast, type-safe  
    - `newInstance()`: Runtime reflection, no-arg constructor only, slower, throws checked exceptions  
    ✅ Use `new` normally; `newInstance()` for frameworks/dynamic loading
  </Card>

  <Card title="Q: `ClassNotFoundException` vs `NoClassDefFoundError`?" icon="error">
    - `ClassNotFoundException`: **Checked Exception** — class not found when explicitly loading (e.g., `Class.forName()`)  
    - `NoClassDefFoundError`: **Error** — class existed at compile but missing at runtime (JVM can't find it when needed)  
    ✅ First is recoverable; second indicates deployment/build issue
  </Card>

  <Card title="Q: Operator precedence: `2 + 3 * 4`?" icon="approve-check">
    **`14`** — `*` has higher precedence than `+`: `2 + (3 * 4) = 2 + 12 = 14`.  
    Use parentheses to override: `(2 + 3) * 4 = 20`.
  </Card>

  <Card title="Q: Evaluation order: `m1(1) + m1(2) * m1(3)`?" icon="information">
    Methods called **left-to-right**: `m1(1)`, then `m1(2)`, then `m1(3)`.  
    Then operations by precedence: `*` before `+`.  
    ✅ Evaluation order ≠ operation order!
  </Card>
</CardGrid>

4. ✅ `new` = compile-time, flexible; `newInstance()` = runtime, no-arg only
5. ✅ `instanceof` = static check; `isInstance()` = dynamic check
6. ✅ `ClassNotFoundException` = your code can catch; `NoClassDefFoundError` = deployment issue
7. ✅ `null instanceof X` → `false` (safe); `null == null` → `true`
8. ✅ Avoid complex compound chains and nested ternaries — prioritize readability
</Aside>
