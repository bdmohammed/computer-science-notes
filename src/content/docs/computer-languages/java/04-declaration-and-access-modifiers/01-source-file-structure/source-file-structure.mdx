## 📁 Java Source File Structure — Complete Guide

> The order of elements in a `.java` file is **strictly enforced** by the compiler. Understanding this structure is critical for writing correct Java code and acing technical interviews.

---

## 🏛️ Valid Source File Structure Diagram

```text
+-----------------------------+
|  1. Package Declaration     |  ← Optional, max 1, MUST be first
+-----------------------------+
|  2. Import Statements       |  ← Optional, many allowed
+-----------------------------+
|  3. Class / Interface       |  ← Required, max 1 public class
+-----------------------------+
|     - Fields                |
|     - Constructors          |
|     - Methods               |
|     - Nested classes        |
+-----------------------------+
```

<Code lang="java" title="Complete valid source file example" code={`// File: com/company/MyApp.java
package com.company;                    // 1. Package (first non-comment statement)

import java.util.ArrayList;             // 2. Imports (any number)
import java.util.List;
import static java.lang.Math.*;         // Static import (Java 5+)

public class MyApp {                    // 3. Public class (name MUST match filename)
    
    private List<String> items;         // Fields
    
    public MyApp() {                    // Constructors
        items = new ArrayList<>();    }
    
    public void addItem(String item) {  // Methods
        items.add(item);
    }
    
    public static void main(String[] args) {
        MyApp app = new MyApp();
        app.addItem("Hello");
        System.out.println(sqrt(16));   // Static import in action
    }
}

// ✅ Additional non-public classes allowed in same file
class Helper {
    static void log(String msg) {
        System.out.println(msg);
    }
}`} />

## ⚠️ Critical Rules — Must Know

<CardGrid>
  <Card title="Rule 1: Package Must Be First" icon="error">
```java
    // ❌ Invalid: import before package
    import java.util.*;
    package com.example;  // ❌ C.E: class, interface, or enum expected
    
    // ✅ Valid: package first (comments allowed before)
    // This is a comment
    package com.example;
    import java.util.*;
```
    **Exception**: Comments and whitespace can appear before `package`.
  </Card>
  
  <Card title="Rule 2: Only ONE Package Statement" icon="error">
```java
    package pack1;
    package pack2;  // ❌ C.E: class, interface, or enum expected
    class A {}
```
    A source file can have **at most one** `package` declaration.
  </Card>

  <Card title="Rule 3: Public Class Name Must Match Filename" icon="error">
```java    // File: MyApp.java
    public class MyApp { }      // ✅ Valid
    
    public class WrongName { }  // ❌ C.E: class WrongName is public, 
                                // should be declared in a file named WrongName.java
```
    If a class is declared `public`, the filename **must exactly match** the class name (case-sensitive).
  </Card>

  <Card title="Rule 4: Only ONE Public Class Per File" icon="error">
```java
    // File: MyApp.java
    public class MyApp { }      // ✅ First public class
    public class Another { }    // ❌ C.E: class Another is public, should be declared separately
```
    A `.java` file can contain **multiple classes**, but **only one** can be `public`.
  </Card>

  <Card title="Rule 5: Non-Public Classes — No Filename Restriction" icon="approve-check">
```java
    // File: AnyName.java (no public class → any filename works)
    class A { public static void main(String[] args) { System.out.println("A"); } }
    class B { public static void main(String[] args) { System.out.println("B"); } }
    class C { public static void main(String[] args) { System.out.println("C"); } }
    
    // Compile: javac AnyName.java → creates A.class, B.class, C.class
    // Run: java A → outputs "A"
    // Run: java B → outputs "B"
    // Run: java C → outputs "C"
```
    When **no class is `public`**, the filename can be anything — no matching required.
  </Card>
</CardGrid>

## 🔍 Compilation & Execution Workflow

### 🧠 How Java Compilation Works

```mermaid
flowchart LR
    A[Source: MyClass.java] --> B[javac MyClass.java]
    B --> C[Bytecode: MyClass.class]
    C --> D[java MyClass]
    D --> E[JVM loads & executes]
```

### 💻 Compile & Run Examples
<Code lang="bash" title="Compiling and running classes" code={`# Case 1: Single class, no package
javac MyApp.java          # Creates MyApp.class
java MyApp                # Runs MyApp.main()

# Case 2: Multiple classes in one file (no public)
# File: AnyName.java contains classes A, B, C
javac AnyName.java        # Creates A.class, B.class, C.class
java A                    # Runs A.main()
java B                    # Runs B.main()

# Case 3: Class without main method
java D                    # 💥 Runtime: NoSuchMethodError: main

# Case 4: .class file missing at runtime
# Delete D.class, then:
java D                    # 💥 Runtime: NoClassDefFoundError: D`} />

<CardGrid>
  <Card title="NoSuchMethodError: main" icon="error">
    **Cause**: The class you're trying to run doesn't contain a valid `main` method.  
    **Fix**: Ensure the class has: `public static void main(String[] args)`
  </Card>
  
  <Card title="NoClassDefFoundError" icon="error">
    **Cause**: The `.class` file existed at compile time but is **missing at runtime** (deleted, wrong classpath, packaging error).  
    **Fix**: Verify the `.class` file exists and is in the classpath.
  </Card>
</CardGrid>

<Aside type="note">
**Key Distinction**:  
- **Compile** the **source file** (`.java`): `javac MyApp.java`  
- **Run** the **class** (`.class`): `java MyApp` (no `.class` extension!)
</Aside>

## 🗂️ Import Statements — Deep Dive

> `import` tells the compiler **where to find** classes you reference — it's a **compile-time shortcut**, not a runtime inclusion.

### ❌ Without Import — Verbose & Error-Prone

<Code lang="java" title="Fully qualified names (no import)" code={`class Test {
    public static void main(String[] args) {
        // Must use full package path every time
        java.util.ArrayList list1 = new java.util.ArrayList();
        java.util.HashMap map = new java.util.HashMap();
        java.util.LinkedList list2 = new java.util.LinkedList();
    }}
// ✅ Compiles, but verbose and hard to read`} />

### ✅ With Import — Clean & Readable

<Code lang="java" title="Import statements simplify code" code={`import java.util.ArrayList;
import java.util.HashMap;
import java.util.LinkedList;

class Test {
    public static void main(String[] args) {
        // Short names — compiler resolves via imports
        ArrayList list1 = new ArrayList();
        HashMap map = new HashMap();
        LinkedList list2 = new LinkedList();
    }
}
// ✅ Same functionality, much cleaner`} />

## 📦 Types of Import Statements

### 1️⃣ Explicit Class Import (Recommended)

> Import **one specific class** by its fully qualified name.

<Code lang="java" title="Explicit imports — clear and safe" code={`import java.util.ArrayList;   // Only ArrayList is imported
import java.util.HashMap;     // Only HashMap is imported
import java.util.LinkedList;  // Only LinkedList is imported

// ✅ Benefits:
// • Reader knows exactly what's used
// • No naming conflicts
// • IDE auto-imports work cleanly
// • Easy to track dependencies`} />

### 2️⃣ Wildcard Import (Use Sparingly)

> Import **all classes** from a package using `*`.

<Code lang="java" title="Wildcard import — convenient but risky" code={`import java.util.*;  // Imports ALL classes in java.util package

// ✅ Pros: Quick for learning, less typing
// ❌ Cons:
// • Unclear which classes are actually used
// • Can cause naming conflicts (see below)
// • Harder for code reviewers to audit

// ⚠️ Important: * does NOT import sub-packages!import java.util.*;           // Imports java.util classes only
// Does NOT import: java.util.concurrent, java.util.stream, etc.
// Need separate import: import java.util.concurrent.*;`} />

### 🔍 Import Validation — Which Are Valid?

<Code lang="java" title="✅ Valid import statements" code={`import java.util.*;              // ✅ Wildcard import
import java.util.ArrayList;      // ✅ Explicit class import
import static java.lang.Math.*;  // ✅ Static wildcard import
import static java.lang.Math.sqrt; // ✅ Static explicit import`} />

<Code lang="java" title="❌ Invalid import statements" code={`import java.util;                 // ❌ C.E: cannot find symbol (need class or *)
import java.util.ArrayList.*;     // ❌ C.E: * not allowed after class name
import static java.lang.Math;     // ❌ C.E: static import needs member, not class
import static java.lang.Math.sqrt.*; // ❌ C.E: * not allowed after method
import static java.lang.Math.sqrt(); // ❌ C.E: () not allowed in import`} />

## ⚠️ Import Naming Conflicts — Ambiguity Trap

### Case 1: Same Class Name in Different Packages

<Code lang="java" title="❌ Ambiguous import: java.util.Date vs java.sql.Date" code={`import java.util.*;
import java.sql.*;

class Test {
    public static void main(String[] args) {
        Date d = new Date();  // ❌ C.E: reference to Date is ambiguous
        // Both java.util.Date and java.sql.Date match!
    }
}
// Error: reference to Date is ambiguous,
// both class java.sql.Date in java.sql and class java.util.Date in java.util match`} />

<Code lang="java" title="✅ Fix: Use explicit import to resolve ambiguity" code={`import java.util.Date;      // ✅ Explicit import takes precedence
import java.sql.*;          // Wildcard import

class Test {
    public static void main(String[] args) {
        Date d = new Date();  // ✅ Uses java.util.Date (explicit wins)
    }
}`} />

### Case 2: Static Import Ambiguity (More Common!)

<Code lang="java" title="❌ Ambiguous static import: MAX_VALUE in Integer vs Byte" code={`import static java.lang.Integer.*;
import static java.lang.Byte.*;

class Test {    public static void main(String[] args) {
        System.out.println(MAX_VALUE);  // ❌ C.E: reference to MAX_VALUE is ambiguous
        // Both Integer.MAX_VALUE (2147483647) and Byte.MAX_VALUE (127) match!
    }
}
// Error: reference to MAX_VALUE is ambiguous,
// both variable MAX_VALUE in java.lang.Integer and variable MAX_VALUE in java.lang.Byte match`} />

<Code lang="java" title="✅ Fix: Explicit static import or use class prefix" code={`// Option 1: Explicit static import (explicit wins over wildcard)
import static java.lang.Integer.MAX_VALUE;
import static java.lang.Byte.*;
// MAX_VALUE now resolves to Integer.MAX_VALUE ✅

// Option 2: Use class prefix (no static import needed)
System.out.println(Integer.MAX_VALUE);  // ✅ Clear and unambiguous
System.out.println(Byte.MAX_VALUE);     // ✅ Also clear`} />

### 🧠 Import Resolution Precedence Order

When the compiler resolves a class or static member name, it checks in this **strict order**:

1.  **Explicit class import** (`import java.util.Date;`)
2.  **Classes in current working directory** (default package)
3.  **Wildcard import** (`import java.util.*;`)

<Code lang="java" title="Precedence example: explicit beats wildcard" code={`import java.util.Date;   // Explicit import (highest priority)
import java.sql.*;       // Wildcard import (lower priority)

class Test {
    public static void main(String[] args) {
        Date d = new Date();  // ✅ Uses java.util.Date (explicit wins)
    }
}`} />

<Aside type="tip">
**Pro Tip**: Prefer **explicit imports** in production code. They:
- Prevent ambiguity errors
- Make dependencies clear to readers
- Help IDEs with refactoring and auto-complete
- Reduce accidental coupling to unused classes
</Aside>

```c
    // main.c    #include "header.h"  // Preprocessor COPIES header.h content into main.c
```

  </Card>

```java
    // Memory: Only referenced classes load at runtime (dynamic)
    // Loading: Classes load on-demand when first used (lazy loading)
```
    **Key**: `import` is a **compile-time hint** — no runtime overhead.

## ⭐ Java 1.5+ Static Import

> Introduced in Java 5, **static import** lets you use static members (methods/fields) without the class name prefix.

### 🔑 Syntax & Usage

<Code lang="java" title="Without static import — verbose" code={`import java.lang.Math;  // Regular import for the class

class Demo {
    public static void main(String[] args) {
        // Must prefix with class name
        System.out.println(Math.sqrt(16));    // Math. required
        System.out.println(Math.abs(-5));     // Math. required
        System.out.println(Math.PI);          // Math. required
        System.out.println(Math.pow(2, 3));   // Math. required
    }
}`} />

<Code lang="java" title="With static import — concise" code={`// Import specific static members
import static java.lang.Math.sqrt;
import static java.lang.Math.abs;
import static java.lang.Math.PI;
import static java.lang.Math.pow;

// OR import all static members (use cautiously!)
// import static java.lang.Math.*;

class Demo {
    public static void main(String[] args) {
        // No class prefix needed!
        System.out.println(sqrt(16));    // ✅ Clean
        System.out.println(abs(-5));     // ✅ Clean
        System.out.println(PI);          // ✅ Clean
        System.out.println(pow(2, 3));   // ✅ Clean
    }
}`} />

### ⚠️ Static Import Resolution Precedence

When resolving a static member name, the compiler checks in this order:
1.  **Current class static members** (highest priority)
2.  **Explicit static import** (`import static java.lang.Integer.MAX_VALUE;`)
3.  **Wildcard static import** (`import static java.lang.Integer.*;`)

<Code lang="java" title="Precedence example: current class beats import" code={`import static java.lang.Integer.MAX_VALUE;  // Line 2: 2147483647
import static java.lang.Byte.*;              // Line 3: 127

class Test {
    // static int MAX_VALUE = 999;           // Line 1: Uncomment to test
    
    public static void main(String[] args) {
        System.out.println(MAX_VALUE);
    }
}
// If Line 1 uncommented: Output = 999 (current class wins)
// If Line 1 commented: Output = 2147483647 (explicit import wins over wildcard)
// If Lines 1-2 commented: Output = 127 (wildcard import used)
`} />

### 🚫 When NOT to Use Static Import

<Aside type="caution">
**Static import reduces readability** and can create confusion. Avoid it unless:
- Using mathematical constants/methods extensively (`Math.*` in scientific code)
- Writing test assertions (`import static org.junit.Assert.*;`)
- Working with enums (`import static java.time.DayOfWeek.*;`)

**Never use static import** for:
- Common method names that may conflict (`max`, `min`, `valueOf`)
- Constants used in only 1-2 places (just use `Class.CONSTANT`)
- Production code where clarity is paramount
</Aside>

## 🖨️ System.out.println — Full Breakdown

> A classic interview question: "Explain `System.out.println`".

<Code lang="java" title="Decomposing System.out.println" code={`System.out.println("Hello");
// │    │    │
// │    │    └─ println() → method of PrintStream class
// │    └─ out → static field of System class (type: PrintStream)
// └─ System → class in java.lang package (auto-imported)

// Fully expanded (what compiler sees):
java.lang.System.out.println("Hello");

// Types of print methods:System.out.println("Hi");  // Prints + newline (\n)
System.out.print("Hi");    // Prints WITHOUT newline
System.out.printf("Name: %s, Age: %d%n", "Alice", 25);  // Formatted output`} />

### 🔧 Can We Shorten System.out.println?

<Code lang="java" title="Static import for System.out" code={`import static java.lang.System.out;  // Import the static field 'out'

class Demo {
    public static void main(String[] args) {
        out.println("Hello");  // ✅ Works! No 'System.' prefix needed
        out.print("World");    // ✅ Also works
    }
}
// ⚠️ Not recommended — reduces clarity. 'System.out' is idiomatic Java.
`} />

## 📦 Package Statement — Namespaces in Java

> A **package** is a namespace that groups related classes and interfaces, providing modularity and avoiding naming conflicts.

### 🔑 Package Naming Convention

```text
com.icicibank.loan.housingloan.Account
│   │          │        │            ││   │          │        │            └── Class name
│   │          │        └──────────── Submodule (feature)
│   │          └──────────────────── Module (domain)
│   └────────────────────────────── Organization (reverse domain)
```

<Code lang="java" title="Package declaration rules" code={`package com.company.project;  // ✅ Must be FIRST non-comment statement

// Rules:
// 1. Only ONE package statement per file
// 2. Must appear before any import or class declaration
// 3. If absent → class goes to "default package" (avoid in production!)
// 4. Follows directory structure: com/company/project/MyClass.java`} />

### 🗂️ Package → Directory Mapping

```text
Source file: com/company/project/MyClass.java
Content:
    package com.company.project;
    public class MyClass { ... }

Compile command (from project root):
    javac -d . com/company/project/MyClass.java

Result:
    ./com/company/project/MyClass.class  ← Auto-created directory structure
```

<Code lang="bash" title="Compile & run package-based classes" code={`# Compile with -d flag to auto-create directory structure
javac -d . src/com/company/MyClass.java

# Run using FULLY QUALIFIED class name (from project root)
java com.company.MyClass

# ❌ Common mistakes:
cd com/company
java MyClass              # ❌ Won't work — JVM needs full package name
java com.company.MyClass.class  # ❌ Don't add .class extension
java com/company/MyClass  # ❌ Use dots, not slashes
`} />

### 🌐 Default Package — Avoid in Production

<Code lang="java" title="Classes without package declaration" code={`// File: MyClass.java (no package statement)
public class MyClass {
    public static void main(String[] args) {
        System.out.println("In default package");
    }
}
// Compiles: javac MyClass.java
// Runs: java MyClass  ✅ (but only from same directory!)

// ⚠️ Problems with default package:
// • Cannot be imported by classes in named packages
// • Hard to organize large projects
// • Not suitable for JAR distribution
// • Avoid in real-world code!
`} />

<Aside type="tip">
**Best Practice**: Always use named packages following reverse-domain convention:
```java
package com.yourcompany.project.module;
```
This ensures:
- No naming conflicts with third-party libraries
- Clear project structure
- Proper JAR packaging and distribution
</Aside>

## 🏗️ The Big Picture — Complete Source File Rules

```text
┌─────────────────────────────────────────────────────────┐
│              MyClass.java                               │
│                                                         │
│  [Comments]              ← Optional, any number         │
│  package com.company;    ← FIRST (optional, max 1)      │
│                                                         │
│  import java.util.*;     ← SECOND (optional, many)      │
│  import static java.lang.Math.*;                        │
│                                                         │
│  public class MyClass {  ← THIRD (mandatory, max 1 public)
│      // Fields                                          │
│      // Constructors                                    │
│      // Methods                                         │
│      // Nested classes (non-public)                     │
│  }                                                      │
│                                                         │
│  class Helper { }        ← Additional non-public classes│
│  interface MyInterface { }                              │
└─────────────────────────────────────────────────────────┘

ORDER IS STRICT:
package → import → class
Any other order → ❌ COMPILE ERROR
```

### 📋 Quick Rules Reference

| Rule | Description | Example |
|------|-------------|---------|
| **Package first** | Must be first non-comment statement | `package com.example;` before imports |
| **One package** | Max one `package` declaration per file | ❌ Two `package` lines → C.E |
| **Public class = filename** | Public class name must match `.java` filename | `public class MyApp` → `MyApp.java` |
| **One public class** | Only one `public` class per source file | ❌ Two `public` classes → C.E |
| **Non-public classes OK** | Unlimited non-public classes/interfaces allowed | `class Helper {}` ✅ in any file |
| **Default package** | No `package` → class in unnamed default package | Avoid in production code |

## 🎯 Interview Cheat Sheet

<CardGrid>
  <Card title="Q: What is the order of elements in a Java source file?" icon="information">
    **Strict order**: `package` → `import` → `class/interface`  
    Any deviation → Compile Error. Comments can appear anywhere.
  </Card>
  
  <Card title="Q: How many package statements allowed?" icon="approve-check">
    **At most ONE** (or zero for default package). Multiple `package` declarations → Compile Error.
  </Card>

  <Card title="Q: How many public classes per .java file?" icon="approve-check">
    **Exactly ONE** (if any), and the filename **must match** that class name.  
    Unlimited non-public classes are allowed.
  </Card>

  <Card title="Q: Does import increase .class file size?" icon="approve-check">
    **NO** — `import` is purely a **compile-time shortcut**. The bytecode references the same fully qualified names regardless.
  </Card>

  <Card title="Q: Difference between import java.util.* and import java.util.List?" icon="information">
    - `import java.util.*`: Imports all classes in `java.util` (not sub-packages!)  
    - `import java.util.List`: Imports only `List`  
    - **Runtime performance**: Identical — both are compile-time only  
    - **Best practice**: Prefer explicit imports for clarity
  </Card>

  <Card title="Q: What is the default package?" icon="caution">
    When no `package` statement is present, the class belongs to the **unnamed default package**.  
    ⚠️ Classes in default package **cannot be imported** by classes in named packages — avoid in production.
  </Card>

  <Card title="Q: Can we use a class without importing it?" icon="approve-check">
    **YES** — use the **fully qualified name**:
    ```java
    java.util.ArrayList list = new java.util.ArrayList<>();  // No import needed
    ```
    `import` is just a convenience to avoid repetitive typing.
  </Card>

  <Card title="Q: Which package is auto-imported?" icon="approve-check">
    **`java.lang`** — automatically available in every Java source file.  
    That's why `String`, `System`, `Object`, `Math` work without explicit import.
  </Card>

  <Card title="Q: What does javac -d flag do?" icon="rocket">
    **`-d <directory>`** tells the compiler where to place generated `.class` files, **automatically creating the package directory structure**.  
    Example: `javac -d out src/com/example/MyClass.java` → creates `out/com/example/MyClass.class`
  </Card>

  <Card title="Q: Static import introduced in which Java version?" icon="information">
    **Java 1.5 (Java 5)** — along with generics, enums, autoboxing, varargs, and enhanced for-loop.
  </Card>
</CardGrid>

## 🔑 Quick Reference Summary

### Source File Order (Strict!)
```text
1. [Comments]          ← Optional
2. package declaration ← Optional, max 1, MUST be first non-comment
3. import statements   ← Optional, any number
4. class/interface     ← Required, max 1 public (name = filename)
5. additional non-public types ← Optional, any number
```

### Import Types Comparison
| Type | Syntax | Imports | Best For |
|------|--------|---------|----------|
| Explicit | `import java.util.ArrayList;` | One class | Production code, clarity |
| Wildcard | `import java.util.*;` | All classes in package | Quick prototyping, learning |
| Static explicit | `import static java.lang.Math.sqrt;` | One static member | Math-heavy code, tests |
| Static wildcard | `import static java.lang.Math.*;` | All static members | Concise math expressions |

### Compilation & Execution
| Task | Command | Notes |
|------|---------|-------|
| Compile single file | `javac MyClass.java` | Creates `MyClass.class` |
| Compile with packages | `javac -d out src/com/example/*.java` | Auto-creates directory structure |
| Run class | `java com.example.MyClass` | Use fully qualified name, no `.class` |
| Run from default package | `java MyClass` | Only works from same directory |

### Common Errors & Fixes
| Error | Cause | Fix |
|-------|-------|-----|
| `class X is public, should be declared in file named X.java` | Public class name ≠ filename | Rename file or class |
| `reference to Date is ambiguous` | Wildcard imports from packages with same class name | Use explicit import |
| `NoSuchMethodError: main` | Class has no valid `main` method | Add `public static void main(String[] args)` |
| `NoClassDefFoundError: X` | `.class` file missing at runtime | Verify classpath, rebuild, check packaging |

<Aside type="caution">
**Final Checklist for Source Files**:
1. ✅ `package` first (if used), then `import`, then `class`
2. ✅ Only one `public` class, filename matches exactly
3. ✅ Prefer explicit imports over wildcards for clarity
4. ✅ Use static imports sparingly — only for clear, conflict-free cases
5. ✅ Always use named packages in production (avoid default package)
6. ✅ Compile with `-d` to manage package directory structure
7. ✅ Run classes using fully qualified names from project root
</Aside>
