## 🔧 Member Modifiers in Java — Complete Guide

> Member modifiers control **visibility**, **behavior**, and **thread-safety** of fields, methods, and blocks. This guide covers all modifiers with practical examples and interview essentials.

---

## 🌍 public Members

> Accessible from **anywhere** — but the **containing class must also be visible**.

<Code lang="java" title="public member — cross-package access" code={`// File: pack1/A.java
package pack1;
class A {  // ❌ Not public → package-private
    public void methodOne() {
        System.out.println("A.methodOne() executed");
    }
}

// File: pack2/B.java
package pack2;
import pack1.A;  // ❌ C.E: pack1.A is not public in pack1; cannot be accessed from outside package

class B {
    public static void main(String[] args) {
        A a = new A();  // Won't compile — class A not accessible
        a.methodOne();  // Even though methodOne() is public!
    }
}`} />

<Aside type="caution">
**Critical Rule**: Member visibility is **secondary** to class visibility.  
Even if a member is `public`, it cannot be accessed if its **containing class is not accessible** from the current package.
</Aside>

### ✅ Visibility Matrix for public Members
| Access From | public Member (in public class) |
|-------------|--------------------------------|
| Same class | ✅ Yes |
| Same package | ✅ Yes |
| Subclass (any package) | ✅ Yes |
| Any other class (any package) | ✅ Yes |

<Code lang="java" title="Correct: Both class and member public" code={`// File: pack1/A.java
package pack1;
public class A {  // ✅ Class is public
    public void methodOne() {  // ✅ Member is public
        System.out.println("A.methodOne() executed");
    }
}

// File: pack2/B.java
package pack2;
import pack1.A;  // ✅ Now works — A is public

class B {
    public static void main(String[] args) {
        A a = new A();  // ✅ Can instantiate
        a.methodOne();  // ✅ Can call public method
    }
}`} />

## 📦 Default Members (Package-Private)

> When **no access modifier** is specified, members have **package-private** visibility — accessible only within the same package.

<Code lang="java" title="Default member — same-package access only" code={`// File: pack1/A.java
package pack1;
class A {
    void methodOne() {  // No modifier → default/package-private
        System.out.println("methodOne executed");
    }
}

// File: pack1/B.java — SAME package ✅
package pack1;
class B {
    public static void main(String[] args) {
        A a = new A();
        a.methodOne();  // ✅ Same package → accessible
    }
}

// File: pack2/C.java — DIFFERENT package ❌package pack2;
import pack1.A;  // ❌ C.E: A is not public in pack1
class C {
    public static void main(String[] args) {
        A a = new A();
        a.methodOne();  // Won't compile
    }
}`} />

### ✅ Visibility Matrix for Default Members

| Access From | Default Member |
|-------------|---------------|
| Same class | ✅ Yes |
| Same package | ✅ Yes |
| Subclass (different package) | ❌ No |
| Other package (non-subclass) | ❌ No |

<Aside type="tip">
**Best Practice**: Use default access for **internal helper methods/fields** that shouldn't be part of your public API. This encapsulates implementation details within the package.
</Aside>

## 🔒 private Members

> Accessible **only within the same class** — maximum encapsulation.

<Code lang="java" title="private member — class-only access" code={`class BankAccount {
    private double balance;       // ✅ Accessible only in BankAccount
    private void deduct() { }     // ✅ Accessible only in BankAccount
    
    public void withdraw(double amount) {
        if (balance >= amount) {
            deduct();  // ✅ Can call private method from same class
            balance -= amount;
        }
    }
}

// ❌ Outside the class — cannot access private members:
class User {
    void tryAccess() {
        BankAccount acc = new BankAccount();
        // acc.balance = 100;  // ❌ C.E: balance has private access
        // acc.deduct();       // ❌ C.E: deduct() has private access
    }
}`} />

### 🔑 Key Rules for private
<CardGrid>
  <Card title="NOT Inherited by Subclasses" icon="error">
```java
    class Parent {
        private int secret = 42;
        private void showSecret() { System.out.println(secret); }
    }
    
    class Child extends Parent {
        void access() {
            // System.out.println(secret);      // ❌ C.E: secret has private access
            // showSecret();                     // ❌ C.E: showSecret() has private access
        }
    }
```
    **Why**: `private` members are not part of the subclass's namespace — they're completely hidden.
  </Card>
  
  <Card title="Illegal: private + abstract" icon="error">
```java
    abstract class Shape {
        private abstract void draw();  // ❌ C.E: illegal combination
        // Reason: abstract requires override; private prevents visibility to child
    }
```
  </Card>
</CardGrid>

### ✅ Visibility Matrix for private Members

| Access From | private Member |
|-------------|---------------|
| Same class | ✅ Yes |
| Same package (different class) | ❌ No |
| Subclass (any package) | ❌ No |
| Other package | ❌ No |

## 🛡️ protected Members

> Accessible within **same package** OR by **subclasses in any package**.

<Code lang="java" title="protected member — inheritance-friendly access" code={`// File: pack1/A.java
package pack1;
public class A {
    protected void methodOne() {
        System.out.println("methodOne executed");
    }}

// File: pack1/B.java — SAME package ✅
package pack1;
class B extends A {
    public static void main(String[] args) {
        A a = new A();
        a.methodOne();  // ✅ Same package → accessible via parent ref
        
        B b = new B();
        b.methodOne();  // ✅ Via child ref
        
        A a1 = new B();
        a1.methodOne(); // ✅ Via parent ref to child object
    }
}`} />

<Code lang="java" title="Different package — only via inheritance ✅" code={`// File: pack2/C.java
package pack2;
import pack1.A;

public class C extends A {  // ✅ Subclass in different package
    public static void main(String[] args) {
        // ❌ Cannot access via parent reference from different package:
        // A a = new A();
        // a.methodOne();  // ❌ C.E: methodOne() has protected access in pack1.A
        
        // ✅ Can access via child reference (inheritance):
        C c = new C();
        c.methodOne();  // ✅ Works — accessed through inheritance
        
        // ✅ Can access via parent ref to child object:
        // A a1 = new C();
        // a1.methodOne();  // ❌ Still C.E — reference type is A, not C
    }
    
    void show() {
        System.out.println(name);  // ✅ If 'name' is protected field in A
    }
}`} />

### ⚠️ Protected Access Nuance — Reference Type Matters

<Code lang="java" title="Key rule: protected access depends on REFERENCE type" code={`package zoo;
import animals.Animal;  // Animal has protected String name;

class Dog extends Animal {
    void test() {
        // ✅ Access via THIS or subclass reference:
        System.out.println(this.name);  // ✅ Works        System.out.println(name);       // ✅ Works (implicit this)
        
        Dog d = new Dog();
        System.out.println(d.name);     // ✅ Works — reference is Dog (subclass)
        
        // ❌ Cannot access via parent reference from different package:
        Animal a = new Animal();
        // System.out.println(a.name);  // ❌ C.E: protected access
        
        Animal a2 = new Dog();
        // System.out.println(a2.name); // ❌ C.E: reference type is Animal
    }
}`} />

### ✅ Visibility Matrix for protected Members

| Access From | protected Member |
|-------------|-----------------|
| Same class | ✅ Yes |
| Same package (any class) | ✅ Yes |
| Subclass (different package) | ✅ Yes — **but only via subclass/this reference** |
| Other package (non-subclass) | ❌ No |

<Aside type="tip">
**Memory Trick**: `protected` = "package + protected by inheritance".  
Think: *"Same package friends OR family (subclasses) can access — but family must prove their lineage (use subclass reference)."*
</Aside>

## 🔐 final Modifier — Variables

> Once assigned, the value **cannot be changed**.

### 🔹 final Instance Variables

<CardGrid>
  <Card title="Rule: Must Initialize Before Constructor Completes" icon="approve-check">
```java
    class Student {
        final int MAX_MARKS;  // ❌ Uninitialized → C.E if not initialized
        
        // ✅ Option 1: At declaration
        final int OPTION1 = 100;
        
        // ✅ Option 2: In instance initializer block
        {
            MAX_MARKS = 100;
        }
        
        // ✅ Option 3: In constructor(s)
        Student() {
            // MAX_MARKS = 100;  // If not already set above
        }
        
        Student(int marks) {
            MAX_MARKS = marks;  // ✅ Can set based on constructor param
        }
    }
```
  </Card>
  
  <Card title="Each Object Can Have Different final Value" icon="information">
```java
    class Config {
        final String API_KEY;  // final but not static
        
        Config(String key) {
            API_KEY = key;  // Set per instance
        }
    }    
    Config prod = new Config("prod-key-123");
    Config dev = new Config("dev-key-456");
    // Each object has its own final API_KEY value ✅
```
  </Card>

  <Card title="❌ Cannot Reassign After Initialization" icon="error">
```java
    class Demo {
        final int x = 10;
        
        void change() {
            x = 20;  // ❌ C.E: cannot assign to final variable x
        }
    }
```
  </Card>
</CardGrid>

### 🔹 final Static Variables

<CardGrid>
  <Card title="Rule: Must Initialize Before Class Loading Completes" icon="approve-check">
```java
    class Config {
        static final int MAX_SIZE;  // ❌ Uninitialized → C.E
        
        // ✅ Option 1: At declaration
        static final int OPTION1 = 100;
        
        // ✅ Option 2: In static initializer block
        static {
            MAX_SIZE = 100;
        }
        
        // ❌ NOT allowed:
        // - Instance initializer block (runs per object, too late)
        // - Constructor (static belongs to class, not instance)
        // - Regular methods (no guarantee of execution timing)
    }
```
  </Card>
  
  <Card title="Classic: static final Constants" icon="rocket">
```java
    public class MathConstants {
        public static final double PI = 3.14159265358979;
        public static final double E = 2.718281828459045;
                // Usage: MathConstants.PI (no instantiation needed)
    }
    
    // Common pattern in production:
    public class AppConstants {
        public static final String APP_NAME = "MyApp";
        public static final int MAX_RETRY = 3;
        public static final long TIMEOUT_MS = 30_000L;
    }
```
  </Card>
</CardGrid>

### 🔹 final Local Variables & Parameters

<Code lang="java" title="final local variable rules" code={`void method() {
    // ✅ Can declare without immediate init (unlike instance/static final)
    final int y;
    y = 50;        // First assignment ✅
    // y = 60;     // ❌ C.E: cannot assign to final variable y (second assignment)
    
    // ✅ final parameter — cannot reassign inside method
    void display(final int x) {
        // x = 100;  // ❌ C.E: cannot assign to final parameter x
        System.out.println(x);  // ✅ Reading is allowed
    }
}`} />

### 🟡 Effectively Final (Java 8+) — Lambda Requirement

<Code lang="java" title="Effectively final for lambdas/anonymous classes" code={`// Java 8+: Variables used in lambdas must be final or effectively final
int x = 10;  // Not declared final, but never reassigned → effectively final

Runnable r = () -> System.out.println(x);  // ✅ Works

// ❌ If you reassign, it's no longer effectively final:
int y = 20;
y = 30;  // Now y is NOT effectively final
// Runnable r2 = () -> System.out.println(y);  // ❌ C.E: variable must be final or effectively final

// ✅ Fix: Don't reassign, or declare explicit final:
final int z = 40;
Runnable r3 = () -> System.out.println(z);  // ✅
`} />

<Aside type="tip">
**Why effectively final?** Lambdas capture variables by value, not reference. If the variable could change after the lambda is created, it would create confusing semantics. The compiler enforces immutability for safety.
</Aside>

## ⚡ static Modifier

> Belongs to the **class**, not instances — shared across all objects.

### 🔑 static Member Rules

<CardGrid>
  <Card title="Shared Across All Instances" icon="information">
```java
    class Counter {
        static int count = 0;  // One copy shared by all objects
        int id;                 // Separate copy per object
        
        Counter() {
            count++;  // Increment shared counter
            id = count;
        }
    }
    
    Counter c1 = new Counter();  // count=1, c1.id=1
    Counter c2 = new Counter();  // count=2, c2.id=2
    Counter c3 = new Counter();  // count=3, c3.id=3
    
    System.out.println(Counter.count);  // 3 (access via class name ✅)
    System.out.println(c1.count);       // 3 (also works, but less clear ⚠️)
```
  </Card>
  
  <Card title="Cannot Access Instance Members Directly" icon="error">
```java
    class Test {
        int instanceVar = 10;
        static int staticVar = 20;
        
        static void staticMethod() {
            System.out.println(staticVar);  // ✅ Can access static
            // System.out.println(instanceVar);  // ❌ C.E: non-static variable cannot be referenced from static context
            
            // ✅ Fix: Access via object reference
            Test t = new Test();
            System.out.println(t.instanceVar);  // ✅
        }
        
        void instanceMethod() {
            System.out.println(instanceVar);  // ✅
            System.out.println(staticVar);    // ✅ (static accessible from instance)
        }
    }
```  </Card>

  <Card title="No this/super in static Context" icon="error">
```java
    class Demo {
        static void show() {
            // System.out.println(this);  // ❌ C.E: non-static variable this cannot be referenced from static context
            // super.toString();           // ❌ C.E: same reason
        }
    }
```
  </Card>
</CardGrid>

### 🔄 Execution Order: static → instance → constructor

<Code lang="java" title="Initialization order proof" code={`class Demo {
    static {
        System.out.println("1. Static block");  // Runs ONCE at class loading
    }
    
    {
        System.out.println("2. Instance block");  // Runs per object, before constructor
    }
    
    Demo() {
        System.out.println("3. Constructor");  // Runs per object
    }
    
    public static void main(String[] args) {
        System.out.println("main() starts");
        new Demo();  // First object
        new Demo();  // Second object
    }
}

/* Output:
1. Static block      ← Once, when class loads
main() starts
2. Instance block    ← First object
3. Constructor       ← First object
2. Instance block    ← Second object  
3. Constructor       ← Second object
*/`} />

### 🧠 static Methods — Inheritance & Overloading

<Code lang="java" title="static methods use method hiding, not overriding" code={`class Parent {
    static void show() {
        System.out.println("Parent.show()");    }
    
    static void main(String[] args) {
        System.out.println("Parent.main()");
    }
}

class Child extends Parent {
    static void show() {  // Hides Parent.show(), does NOT override
        System.out.println("Child.show()");
    }
    
    static void main(String[] args) {  // Hides Parent.main()
        System.out.println("Child.main()");
    }
}

// Execution depends on REFERENCE type, not object type:
Parent p = new Child();
p.show();  // "Parent.show()" ← Reference type is Parent

Child c = new Child();
c.show();  // "Child.show()" ← Reference type is Child

// JVM always calls main() based on the class you execute:
// java Parent → Parent.main()
// java Child  → Child.main() (hides parent's main)
`} />

<Aside type="note">
**Key Distinction**:  
- **Overriding** (instance methods): Runtime polymorphism — method called depends on **object type**  
- **Hiding** (static methods): Compile-time binding — method called depends on **reference type**
</Aside>

## 🔧 native Modifier

> Marks a method whose implementation is in **native code** (C/C++) via JNI.

<Code lang="java" title="native method declaration" code={`class FileSystem {
    // Native method — NO body in Java, ends with semicolon
    public native void readFile(String path);
    
    // Static initializer to load native library
    static {
        System.loadLibrary("NativeFileIO");  // Loads libNativeFileIO.so or NativeFileIO.dll
    }
}
// Usage:
FileSystem fs = new FileSystem();
fs.readFile("/data/file.txt");  // Calls native C/C++ implementation
`} />

### 🔑 native Method Rules

<CardGrid>
  <Card title="No Java Body — Semicolon Only" icon="approve-check">
```java
    public native void method();  // ✅ Valid — ends with semicolon
    // public native void method() { }  // ❌ C.E: native methods cannot have a body
```
  </Card>
  
  <Card title="Illegal Combinations" icon="error">
```java
    // ❌ abstract + native: both imply "no body" → contradiction
    abstract native void method();  // ❌ C.E
    
    // ❌ strictfp + native: native code may not follow IEEE 754
    strictfp native double calc();  // ❌ C.E (Java spec prohibits)
    
    // ✅ static + native: allowed
    static native long currentTimeMillis();  // Like System.currentTimeMillis()
```
  </Card>

  <Card title="Real JDK Examples" icon="information">
```java
    // These JDK methods are native under the hood:
    System.currentTimeMillis();  // OS system call
    Object.hashCode();           // JVM object header access
    Thread.sleep(long ms);       // OS thread suspension
    FileInputStream.read();      // OS file I/O
```
  </Card>
</CardGrid>

### ⚖️ Pros & Cons of native

| Advantage | Disadvantage |
|-----------|--------------|
| ✅ Performance: Direct hardware/OS access | ❌ Breaks "write once, run anywhere" |
| ✅ Reuse legacy C/C++ libraries | ❌ Platform-specific binaries (.dll, .so) |
| ✅ Access low-level system features | ❌ Harder to debug, maintain, test |
| ✅ Critical sections in native code | ❌ Security risks (native code bypasses JVM safety) |

<Aside type="caution">**Best Practice**: Use `native` only when absolutely necessary (e.g., hardware drivers, performance-critical math libraries). Prefer pure Java for portability and maintainability.
</Aside>

## 🔀 synchronized Modifier

> Ensures **mutual exclusion** — only one thread executes the synchronized code at a time.

### 🔑 synchronized Rules

<CardGrid>
  <Card title="Applies to Methods and Blocks Only" icon="approve-check">
    ```java
    class BankAccount {
        private int balance = 1000;
        
        // ✅ Synchronized method — locks on 'this' object
        public synchronized void withdraw(int amount) {
            if (balance >= amount) {
                balance -= amount;
            }
        }
        
        // ✅ Synchronized block — finer-grained locking
        public void deposit(int amount) {
            synchronized (this) {  // Lock on this object
                balance += amount;
            }
            // Other non-critical code can run concurrently
        }
        
        // ❌ Cannot synchronize constructors (not inherited, single-threaded creation)
        // public synchronized BankAccount() { }  // ❌ C.E: illegal modifier for constructor
        
        // ❌ Cannot synchronize classes or variables
        // synchronized class X { }  // ❌ C.E
        // synchronized int x;       // ❌ C.E
    }
    ```
  </Card>
  
  <Card title="Illegal: abstract + synchronized" icon="error">
    ```java
    abstract class Task {
        // ❌ abstract has no body; synchronized needs body to lock
        abstract synchronized void execute();  // ❌ C.E
    }
    ```
  </Card>
</CardGrid>

### ⚠️ synchronized vs volatile — Key Difference

<Code lang="java" title="synchronized guarantees atomicity + visibility" code={`class Counter {
    private int count = 0;
    
    // ✅ synchronized: atomic increment + visibility guarantee
    public synchronized void increment() {
        count++;  // Read-modify-write is atomic here
    }
    
    public synchronized int getCount() {
        return count;  // Always reads latest value
    }
}

// ❌ volatile alone does NOT make increment atomic:
class VolatileCounter {
    private volatile int count = 0;
    
    public void increment() {
        count++;  // ❌ NOT atomic! Still 3 operations: read, increment, write
        // Two threads could read same value → lost update
    }
}
`} />

<Aside type="tip">
**When to Use Which**:
- `synchronized`: Need **atomicity + visibility** (e.g., counter, shared state updates)
- `volatile`: Need **visibility only** for simple flags (e.g., `volatile boolean running`)
- `AtomicInteger/Long`: Lock-free atomic operations (often better than synchronized for counters)
</Aside>

## 💾 transient Modifier

> Field is **skipped during serialization** — not saved to byte stream.

<Code lang="java" title="transient field — excluded from serialization" code={`import java.io.Serializable;

class User implements Serializable {
    String username = "alice";              // ✅ Serialized
    transient String password = "secret123"; // ❌ NOT serialized
    transient int sessionToken;             // ❌ NOT serialized
    static String dbUrl = "jdbc:mysql://..."; // ❌ static never serialized anyway
}

// After serialize → deserialize:
User original = new User();
// Serialize to file/network...
User restored = deserialize();  // Pseudocode

System.out.println(restored.username);    // "alice" ✅
System.out.println(restored.password);    // null ❌ (transient reset to default!)
System.out.println(restored.sessionToken); // 0 ❌ (transient reset to default)
`} />

### 🔑 transient Rules

<CardGrid>
  <Card title="Only for Instance Variables" icon="approve-check">
    ```java
    class Data {
        transient int temp;           // ✅ Valid
        // transient static int x;    // ⚠️ Compiles but pointless — static not serialized anyway
        // transient final int y = 5; // ⚠️ Compiles but confusing — final serialized by value, transient says skip
    }
    ```
  </Card>
  
  <Card title="Resets to Default Value on Deserialization" icon="information">
    | Original Type | transient Default After Deserialize |
    |--------------|-------------------------------------|
    | `int`, `short`, `byte`, `long` | `0` |
    | `float`, `double` | `0.0` |
    | `char` | `'\u0000'` |
    | `boolean` | `false` |
    | Object reference | `null` |
  </Card>
</CardGrid>

<Aside type="tip">
**Use Cases for transient**:
- Passwords, tokens, API keys (security)
- Cached/computed values (can be recalculated)
- Thread-local state (not meaningful after deserialization)
- Database connections, file handles (resource handles not serializable)
</Aside>

## 💨 volatile Modifier

> Guarantees **visibility** of variable changes across threads — writes go directly to main memory.

<Code lang="java" title="volatile ensures visibility across threads" code={`class Server {
    // volatile: writes by one thread are immediately visible to others
    private volatile boolean running = true;
    
    void stop() {
        running = false;  // Thread A: write to main memory
    }
    
    void run() {
        while (running) {  // Thread B: always reads from main memory
            handleRequest();  // Never sees stale cached value ✅
        }
    }
}
`} />

### 🔑 volatile Rules & Limitations

<CardGrid>
  <Card title="Only for Variables — Not Methods/Classes" icon="approve-check">
    ```java
    class Config {
        volatile boolean updated;  // ✅ Valid
        // volatile void update() { }  // ❌ C.E: volatile not allowed on methods
        // volatile class X { }        // ❌ C.E: volatile not allowed on classes
    }
    ```
  </Card>
  
  <Card title="Illegal: final + volatile" icon="error">
    ```java
    class Data {
        // ❌ final = never changes; volatile = changes frequently → contradiction
        final volatile int x = 10;  // ❌ C.E: illegal combination
    }
    ```
  </Card>

  <Card title="volatile ≠ Atomic — Compound Operations Still Need Sync" icon="error">
    ```java
    volatile int count = 0;
    
    void increment() {
        count++;  // ❌ NOT atomic! Still 3 steps:
        // 1. Read count from main memory
        // 2. Increment locally
        // 3. Write back to main memory
        // Two threads could interleave → lost update
    }
    
    // ✅ Fix: Use AtomicInteger for atomic increment
    AtomicInteger atomicCount = new AtomicInteger(0);
    void incrementSafe() {
        atomicCount.incrementAndGet();  // ✅ Atomic operation
    }
    ```
  </Card>
</CardGrid>

## ❌ Illegal Modifier Combinations Summary

<CardGrid>
  <Card title="Method-Level Illegal Combinations" icon="error">
    ```java
    abstract final void m1();        // ❌ abstract=override, final=no override
    abstract private void m2();      // ❌ abstract=visible to child, private=hidden
    abstract static void m3();       // ❌ abstract=runtime poly, static=compile-time
    abstract native void m4();       // ❌ both mean "no Java body" → contradiction
    abstract synchronized void m5(); // ❌ abstract=no body, synchronized=needs body
    abstract strictfp void m6();     // ❌ abstract=no body, strictfp=needs body to be strict about

## 🎯 Interview Cheat Sheet

<CardGrid>
  <Card title="Q: Where can final instance variable be initialized?" icon="approve-check">
    **Exactly 3 places**:
    1. At declaration: `final int x = 10;`
    2. In instance initializer block: `{ x = 10; }`
    3. In constructor(s): `MyClass() { x = 10; }`  
    ❌ Cannot leave uninitialized → compile error.
  </Card>
  
  <Card title="Q: Where can final static variable be initialized?" icon="approve-check">
    **Exactly 2 places**:
    1. At declaration: `static final int X = 10;`
    2. In static initializer block: `static { X = 10; }`  
    ❌ Cannot use instance block/constructor — static belongs to class, not instance.
  </Card>

  <Card title="Q: Can volatile guarantee atomicity?" icon="error">
    **NO ❌** — `volatile` only guarantees **visibility**, not atomicity.  
    `volatile int x; x++;` is still **3 operations** (read, increment, write) → not thread-safe.  
    ✅ Use `AtomicInteger`, `synchronized`, or `Lock` for atomicity.
  </Card>

  <Card title="Q: What happens to transient fields after deserialization?" icon="information">
    They are **reset to default values**:
    - Numeric types → `0` or `0.0`
    - `boolean` → `false`
    - `char` → `'\u0000'`
    - Object references → `null`  
    ✅ Use this for sensitive data (passwords) or recomputable caches.
  </Card>

  <Card title="Q: Can static method access instance variables?" icon="error">
    **NO ❌** — static methods belong to the class, not instances.  
    ✅ Fix: Access via object reference: `obj.instanceVar`  
    ✅ Instance methods CAN access static variables (class state is always available).
  </Card>

  <Card title="Q: Difference between overriding and hiding?" icon="information">
    - **Overriding** (instance methods): Runtime polymorphism — method called depends on **object type**  
    - **Hiding** (static methods): Compile-time binding — method called depends on **reference type**  
    Example: `Parent p = new Child(); p.show();` → calls `Parent.show()` if static, `Child.show()` if instance.
  </Card>

  <Card title="Q: What is effectively final (Java 8+)?" icon="approve-check">
    A variable **not declared `final` but never reassigned** after initialization.  
    ✅ Can be used in lambdas/anonymous classes (compiler treats as final).  
    ❌ If reassigned anywhere, it's no longer effectively final → lambda usage fails.
  </Card>

  <Card title="Q: protected access in different package — what's the catch?" icon="caution">
    Protected members are accessible in subclasses **only via subclass/this reference**, not parent reference.  
    ```java
    // In subclass in different package:
    this.protectedField;      // ✅
    protectedField;           // ✅ (implicit this)
    Subclass s; s.protectedField;  // ✅
    Parent p; p.protectedField;    // ❌ C.E — reference type is Parent
    ```
  </Card>
</CardGrid>

## 🔑 Quick Reference Summary

### Access Modifier Visibility

| Modifier | Same Class | Same Package | Subclass (Diff Package) | World |
|----------|-----------|--------------|------------------------|-------|
| `private` | ✅ | ❌ | ❌ | ❌ |
| `default` | ✅ | ✅ | ❌ | ❌ |
| `protected` | ✅ | ✅ | ✅* | ❌ |
| `public` | ✅ | ✅ | ✅ | ✅ |

<small>* protected in subclass requires reference to be subclass type or `this`.</small>

### final Initialization Requirements

| Variable Type | Must Initialize In | Default Value if Uninitialized |
|--------------|-------------------|-------------------------------|
| Instance (non-final) | Anywhere (or never) | JVM provides (`0`, `false`, `null`) |
| Instance `final` | Declaration, instance block, or constructor | ❌ Compile Error |
| Static (non-final) | Anywhere (or never) | JVM provides |
| Static `final` | Declaration or static block | ❌ Compile Error |
| Local (any) | Before first use | ❌ Compile Error |
| Parameter `final` | At declaration (method signature) | N/A |

### Thread-Safety Modifiers

| Modifier | Guarantees | Use Case | Caveat |
|----------|-----------|----------|--------|
| `synchronized` | Atomicity + Visibility + Mutual Exclusion | Compound operations, shared state | Locking overhead |
| `volatile` | Visibility only | Simple flags, status indicators | ❌ Not atomic for compound ops |
| `final` | Safe publication (for immutable objects) | Immutable configuration, constants | Only for references — content may still be mutable |

### Common Compile Errors & Fixes

| Error Message | Likely Cause | Fix |
|--------------|--------------|-----|
| `variable x might not have been initialized` | `final` instance/static variable not initialized | Initialize in allowed location (declaration/block/constructor) |
| `non-static variable x cannot be referenced from static context` | Accessing instance field from static method | Access via object reference: `obj.x` |
| `methodOne() has protected access` | Accessing protected member via parent ref in different package | Use subclass reference or move access into subclass |
| `illegal combination of modifiers: abstract and final` | Contradictory modifiers | Choose one: `abstract` OR `final` |
| `native methods cannot have a body` | Added `{ }` to native method | Remove body, end with `;` |
| `synchronized is not allowed here` | Used on constructor/class/variable | Apply only to methods or blocks |

<Aside type="caution">
**Final Checklist for Member Design**:
1. ✅ Start with **least access** needed (`private` → expand only if necessary)
2. ✅ Use `final` for **immutability** — fields, parameters, local variables
3. ✅ Initialize `final` fields in **allowed locations only** (compile enforces this)
4. ✅ Use `static` for **class-level state** — remember: no `this`, can't access instance members directly
5. ✅ Use `synchronized` for **atomicity**, `volatile` for **visibility only**
6. ✅ Use `transient` for **sensitive or recomputable** fields in serializable classes
7. ✅ Avoid illegal combinations: `abstract + final`, `final + volatile`, etc.
8. ✅ Prefer `java.util.concurrent` utilities over manual `synchronized`/`volatile` when possible
</Aside>
